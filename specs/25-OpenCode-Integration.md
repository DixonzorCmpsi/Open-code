# Spec 25: OpenCode Integration ŌĆö Claw's Execution OS

**Status:** ACTIVE ŌĆö Supersedes `specs/07-OpenClaw-OS.md`
**Replaces:** Custom `openclaw-gateway` TypeScript runtime
**Introduced:** Phase 2 migration to OpenCode as the execution OS

---

## 0. Executive Summary

Claw's execution backend is **OpenCode** (`opencode.ai`) ŌĆö an open-source AI coding agent used by 5M+ developers monthly. Instead of maintaining a custom TypeScript gateway server, Claw compiles `.claw` source files into OpenCode's native configuration format. OpenCode handles:

- Model/provider invocation (75+ LLM providers)
- Session and memory management
- Tool execution and MCP server hosting
- Permission sandboxing
- LSP integration
- Multi-agent parallelism
- Shareable sessions and streaming

Claw's value is the **typed, statically-verified orchestration language on top** of OpenCode. The DSL remains unchanged. Only the compilation target and runtime changes.

## 1. Architecture Overview

### 1.1 Before (Custom Gateway ŌĆö RETIRED)

```
.claw ŌåÆ clawc ŌåÆ TS/Python SDK ŌåÆ openclaw-gateway (custom OS) ŌåÆ LLM APIs
```

### 1.2 After (OpenCode OS ŌĆö CURRENT)

```
.claw ŌåÆ clawc ŌåÆ opencode.json                 ŌåÉ agents.coder.model + mcpServers + contextPaths
                .opencode/commands/*.md        ŌåÉ one per workflow block
                generated/mcp-server.js        ŌåÉ one MCP tool per tool block + agent runner tools
                generated/claw-context.md      ŌåÉ all type/agent/workflow docs for OpenCode context
              ŌåÆ opencode (the OS)
              ŌåÆ LLM APIs (75+ providers)
```

**Note:** `.opencode/agents/*.md` does NOT exist in the OpenCode runtime. OpenCode only supports four built-in agent types (`coder`, `task`, `title`, `summarizer`) configured via `opencode.json`. Named custom agents from `.claw` are implemented via MCP agent-runner tools in `generated/mcp-server.js` (see ┬¦2.3).

### 1.3 Separation of Concerns

| Component | Owner | Responsibility |
|-----------|-------|---------------|
| `.claw` source | Developer | Declare types, clients, tools, agents, workflows |
| `clawc` (Rust compiler) | Claw project | Parse, type-check, emit OpenCode config |
| `opencode.json` | Generated by `clawc` | `agents.coder.model`, `mcpServers`, `contextPaths` |
| `.opencode/commands/*.md` | Generated by `clawc` | Per-workflow slash commands (`/WorkflowName`) |
| `generated/mcp-server.js` | Generated by `clawc` | All `tool` blocks + `agent_<Name>` runner tools |
| `generated/claw-context.md` | Generated by `clawc` | All type schemas, agent prompts, workflow docs |
| OpenCode runtime | OpenCode project | All execution, streaming, session management |

## 2. DSL ŌåÆ OpenCode Mapping

### 2.1 `client` block ŌåÆ OpenCode provider config

**Claw source:**
```claw
client FastClaude {
    provider = "anthropic"
    model = "claude-sonnet-4-6"
    retries = 3
}

client LocalLLM {
    provider = "custom"
    model = "mistral-7b"
    endpoint = env("OLLAMA_URL")
}
```

**Emitted in `opencode.json`:**
```json
{
  "agents": {
    "coder": {
      "model": "claude-4-sonnet"
    }
  }
}
```

**Rules:**
- The first `client` block in the file sets `agents.coder.model`.
- OpenCode model IDs use the format defined in `opencode-schema.json` ŌĆö e.g., `claude-4-sonnet`, `claude-4-opus`, `gpt-4o`. The Claw `model` field value is passed through as-is; the developer must use an OpenCode-valid model ID.
- `provider` and API keys are NOT emitted in `opencode.json`. OpenCode reads them from environment variables (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, etc.) automatically. The `client.provider` field is used only to determine which env var the developer needs to set ŌĆö the compiler prints a hint, not a config field.
- `retries` has no OpenCode equivalent. The compiler emits a warning and does not emit the field.
- `env("VAR")` in `client.endpoint` is noted in `generated/claw-context.md` as a setup instruction. OpenCode does not use `{env:VAR}` substitution in `opencode.json` fields ŌĆö env vars are read by OpenCode at the process level.

### 2.2 `tool` block ŌåÆ MCP server + `opencode.json` registration

**Claw source:**
```claw
tool WebSearch(query: string) -> SearchResult {
    invoke: module("scripts.search").function("run")
}

tool ReadFile(path: string) -> string {
    invoke: module("scripts.fs").function("read")
}
```

**Emitted in `generated/mcp-server.js`:**
All `tool` blocks are compiled into a single MCP server file. See `specs/26-MCP-Server-Generation.md` for the full generation spec.

**Emitted in `opencode.json`:**
```json
{
  "mcpServers": {
    "claw-tools": {
      "command": "node",
      "args": ["generated/mcp-server.js"],
      "type": "stdio"
    }
  }
}
```

**Rules:**
- All tools from a single `.claw` file are bundled into ONE `mcp-server.js`.
- The MCP server key is always `"claw-tools"`. OpenCode starts it as a child process via stdio.
- The correct OpenCode config key is `mcpServers` (NOT `mcp`). The server type is `"stdio"` (NOT `"local"`).
- Tool argument types are translated to JSON Schema for MCP tool registration (TypeBox lowering).
- `invoke: module(...)` becomes the implementation call within the MCP server handler.
- The MCP server also contains `agent_<Name>` runner tools ŌĆö one per `agent` block (see ┬¦2.3).

### 2.3 `agent` block ŌåÆ MCP agent-runner tool + `generated/claw-context.md` section

**OpenCode does NOT support custom named agents via `.opencode/agents/*.md`.** The OpenCode runtime only has four built-in agents: `coder`, `task`, `title`, `summarizer`. All are configured in `opencode.json` under the `agents` key.

Claw implements named multi-agent orchestration via two generated artifacts:

1. **An `agent_<Name>` MCP tool** in `generated/mcp-server.js` ŌĆö when the coder agent calls `agent_Researcher`, the MCP tool handler **calls the LLM provider API directly** (using the Anthropic SDK, Ollama, or other configured provider) with the agent's declared `system_prompt`, `tools`, and `settings`. It manages the tool-call loop internally and returns the typed result. It does **NOT** spawn a child `opencode` process.
2. **A section in `generated/claw-context.md`** ŌĆö documents the agent's role, tools, and constraints so the coder agent understands when and how to invoke it.

**Claw source:**
```claw
agent Researcher {
    client = FastClaude
    system_prompt = "You form exact hypotheses before executing tools."
    tools = [WebSearch, ReadFile]
    settings = {
        max_steps: 5,
        temperature: 0.1
    }
}
```

**Emitted in `generated/mcp-server.js` (agent runner tool):**
```javascript
{
  name: "agent_Researcher",
  description: "Run the Researcher agent. You form exact hypotheses before executing tools. Available tools: WebSearch, ReadFile.",
  inputSchema: {
    type: "object",
    properties: {
      task: { type: "string", description: "The task to perform" }
    },
    required: ["task"]
  }
}

// The agent handler calls the LLM provider API directly ŌĆö no child process spawning.
// It manages the full tool-call loop internally and validates the return type.
async function handleagent_Researcher(args) {
  const { task } = args;
  const Anthropic = (await import("@anthropic-ai/sdk")).default;
  const client = new Anthropic();  // reads ANTHROPIC_API_KEY from env

  const messages = [{ role: "user", content: task }];
  const tools = [WEBSEARCH_TOOL_SCHEMA, READFILE_TOOL_SCHEMA];

  // Tool-call loop: call LLM, dispatch tool calls, repeat until end_turn
  while (true) {
    const response = await client.messages.create({
      model: "claude-4-sonnet",
      system: "You form exact hypotheses before executing tools.",
      messages,
      tools,
      max_tokens: 4096,
    });

    if (response.stop_reason === "end_turn") {
      const text = response.content.find(b => b.type === "text")?.text ?? "";
      // Parse and validate against declared return type (if any)
      return { content: [{ type: "text", text }] };
    }

    // Handle tool calls
    const toolUseBlocks = response.content.filter(b => b.type === "tool_use");
    messages.push({ role: "assistant", content: response.content });
    const toolResults = [];
    for (const toolUse of toolUseBlocks) {
      const handler = HANDLERS[toolUse.name];
      const result = handler ? await handler(toolUse.input) : { error: `Unknown tool: ${toolUse.name}` };
      toolResults.push({ type: "tool_result", tool_use_id: toolUse.id,
                         content: JSON.stringify(result) });
    }
    messages.push({ role: "user", content: toolResults });
  }
}
```

See `specs/26-MCP-Server-Generation.md ┬¦2.5` for the full agent handler generation pattern, including provider routing (Anthropic vs. Ollama) and return type validation.

**Emitted in `generated/claw-context.md` (agent documentation section):**
```markdown
## Agent: Researcher
- Model: claude-4-sonnet
- System prompt: You form exact hypotheses before executing tools.
- Available tools (via MCP): WebSearch, ReadFile
- Max steps: 5
- Invoke via MCP tool: agent_Researcher({ task: "..." })
```

**Rules:**
- Each `agent` block generates one `agent_<Name>` MCP tool in `mcp-server.js`.
- The agent handler calls the LLM provider API **directly** ŌĆö it MUST NOT spawn a child `opencode` process.
- `client` resolves to the provider SDK and model (Anthropic SDK for `provider = "anthropic"`, Ollama HTTP for `provider = "local"`).
- `system_prompt` is passed as the `system` parameter in the LLM API call.
- `settings.max_steps` is enforced as a loop counter in the tool-call loop (abort with error after N steps).
- `extends` is resolved by the compiler before codegen; the agent's final merged properties are what get emitted.
- `tools +=` is resolved at compile time; the final merged tool list is what gets registered in the tool-call loop.
- Return type validation (`require_type`) runs after `end_turn` ŌĆö parse the LLM text as JSON, validate against the declared schema, return MCP error if invalid.

### 2.4 `workflow` block ŌåÆ `.opencode/commands/{Name}.md`

**Claw source:**
```claw
workflow AnalyzeCompetitors(companies: list<string>) -> list<string> {
    let final_reports = []
    for (company in companies) {
        let findings: list<SearchResult> = execute Researcher.run(
            task: "Find latest product announcements for ${company}",
            require_type: list<SearchResult>
        )
        let report: string = execute SeniorResearcher.run(
            task: "Summarize findings",
            context: findings
        )
        final_reports.append(report)
    }
    return final_reports
}
```

**Emitted in `.opencode/commands/AnalyzeCompetitors.md`:**
```markdown
Run the AnalyzeCompetitors workflow.

Companies to analyze: $COMPANIES

Steps:
1. For each company, call agent_Researcher with task: "Find latest product announcements for {company}"
2. For each set of findings, call agent_SeniorResearcher with task: "Summarize findings for {company}"
3. Return all summaries as a list

Expected output: list of report strings ŌĆö one paragraph per company.
```

**Rules:**
- Workflow name maps to the command filename (case-preserved).
- OpenCode command argument substitution uses `$UPPERCASE_PARAM_NAME` (regex: `\$([A-Z][A-Z0-9_]*)`). Each workflow parameter is uppercased: `topic` ŌåÆ `$TOPIC`, `companies` ŌåÆ `$COMPANIES`.
- There is no `agent` frontmatter field in the actual OpenCode commands system. The command body is sent to the `coder` agent. The coder agent uses MCP tools (including `agent_<Name>` runner tools) to execute the workflow.
- The workflow body logic is translated into numbered steps referencing the MCP tool names.
- The return type schema is always documented in the command body.
- Complex control flow (`for`, conditionals) is expressed as natural language steps.

**Multi-parameter workflows** encode all parameters as separate `$UPPERCASE` substitution variables:

```markdown
Run AnalyzeCompetitors.

Companies: $COMPANIES
Analysis depth: $DEPTH
Region: $REGION

Steps: ...

Expected output: list<string>
```

OpenCode prompts the user for each `$VARIABLE` value before running the command. Single-parameter workflows with a primitive `string` type use one `$PARAM_NAME` variable directly.

**Additionally emitted:** A typed TypeScript wrapper in `generated/claw/index.ts` when `--lang opencode` is combined with `--also-ts`, or separately with `--lang ts` (see `specs/06-CodeGen-SDK.md`).

### 2.5 `type` block ŌåÆ TypeScript/Python types + MCP schemas

Type declarations are not emitted as OpenCode configuration. They are used by:
- The TypeScript SDK (`generated/claw/index.ts`) as Zod schemas
- The Python SDK (`generated/claw/__init__.py`) as Pydantic models
- The MCP server (`generated/mcp-server.js`) as JSON Schema for tool input/output validation

### 2.6 `listener` block ŌåÆ OpenCode custom command with trigger

**Claw source:**
```claw
listener OnSlackMessage(event: Events.Slack.Message) {
    if (event.channel == "research-requests") {
        let result = await AnalyzeCompetitors(["OpenAI", "Anthropic"])
        event.reply(result)
    }
}
```

**Mapping:** Listener blocks are compiled into a custom command with a trigger description comment. The actual event subscription is handled by the user's application code calling the OpenCode SDK. This is a Phase 3 feature; listener blocks produce a warning during compilation if `--lang opencode` is used without a configured event adapter.

## 3. `opencode.json` Full Schema

The generated `opencode.json` contains Claw-derived configuration for OpenCode.

**Merge strategy (MUST NOT do a full overwrite):**
- If `opencode.json` does not exist: write fresh with all Claw fields.
- If `opencode.json` exists: read it, update ONLY the Claw-owned fields (`model`, `provider`, `mcp.claw-tools`, `instructions`), and write back. All other fields (`theme`, `keybinds`, `formatter`, `plugin`, `share`, etc.) are preserved unchanged.
- Claw-owned fields are marked with a comment in JSONC: `// managed by clawc ŌĆö do not edit`.

This prevents `claw build` from destroying user-customized OpenCode settings.

```json
{
  "model": "ollama/qwen2.5:14b",
  "provider": {
    "ollama": {
      "api": "http://localhost:11434/v1",
      "models": { "qwen2.5:14b": {} }
    }
  },
  "mcp": {
    "claw-tools": {
      "command": ["/opt/homebrew/bin/node", "generated/mcp-server.js"],
      "type": "local"
    }
  },
  "instructions": [
    "generated/claw-context.md"
  ]
}
```

**Fields (verified against OpenCode 1.2.27 runtime behaviour):**

| Field | Source | Description |
| ----- | ------ | ----------- |
| `model` | First `client` block | Top-level model ID. For `provider = "local"`: `"ollama/<model-id>"`. For cloud: model ID as-is (e.g., `"claude-4-sonnet"`). |
| `provider.ollama` | First `client` block when `provider = "local"` | Ollama connection config. `api` defaults to `http://localhost:11434/v1`. `models` registers the model. Omitted for cloud providers. |
| `mcp.claw-tools` | All `tool` + `agent` blocks | Starts `generated/mcp-server.js` as a local child process. Key is `mcp` (NOT `mcpServers`). Type is `"local"` (NOT `"stdio"`). Command is an array: `[node-binary-path, "generated/mcp-server.js"]`. |
| `instructions` | Always present | Array of paths OpenCode injects as context. Points to `generated/claw-context.md`. Field is `instructions` (NOT `contextPaths`). |

**ŌÜĀ Common mistakes that cause "Unrecognized keys" rejection by OpenCode 1.2.27:**

- Using `mcpServers` instead of `mcp`
- Using `type: "stdio"` instead of `type: "local"`
- Using `agents.coder.model` instead of top-level `model`
- Using `contextPaths` instead of `instructions`

**What is NOT emitted (and why):**

| Omitted field | Reason |
| ------------- | ------ |
| `providers.*` | API keys come from env vars (`ANTHROPIC_API_KEY` etc.). OpenCode detects them automatically. |
| `agents.task`, `agents.title`, `agents.summarizer` | Not overridden by Claw ŌĆö OpenCode defaults are correct. |
| `autoCompact` | OpenCode default (`true`) is correct for all Claw workflows. |
| `tui.theme` | User preference, never Claw-managed. |

**Merge strategy:** Read existing `opencode.json`, update only `agents.coder.model`, `mcpServers.claw-tools`, and `contextPaths`. All other fields are preserved unchanged.

## 4. `generated/claw-context.md` (Project Context Document)

The compiler generates a project context document at `generated/claw-context.md` (NOT `AGENTS.md` ŌĆö to avoid colliding with user-authored project docs). It is referenced from `opencode.json` via the `contextPaths` array (NOT `instructions` ŌĆö that field does not exist in the actual OpenCode schema).

If a project-level `AGENTS.md` already exists, the compiler prepends it: `"contextPaths": ["AGENTS.md", "generated/claw-context.md"]`.

The compiler generates a project context document for OpenCode that describes the `.claw` project:

```markdown
# Claw Project Context

This project uses the Claw DSL for deterministic multi-agent orchestration.

## Types
- `SearchResult`: url (string), confidence_score (float), snippet (string), tags (list<string>)

## Agents
- `Researcher`: Uses FastClaude (anthropic/claude-sonnet-4-6). Tools: WebSearch, ReadFile.
- `SeniorResearcher`: Extends Researcher. Additional tools: WriteFile.

## Workflows
- `AnalyzeCompetitors(companies: list<string>) -> list<string>`: Fans out to Researcher per company, aggregates via SeniorResearcher.

## Tools (MCP)
- `WebSearch(query: string) -> SearchResult`: Invokes scripts.search.run
- `ReadFile(path: string) -> string`: Invokes scripts.fs.read

## Constraints
- All agent outputs are validated against their declared return types.
- Schema degradation (all-zero outputs) causes automatic retry (max 3).
- Session recursion depth is bounded at 10 levels.
```

## 5. OpenCode Runtime Capabilities

These OpenCode features replace what was previously handled by `openclaw-gateway`:

| OpenClaw Gateway Feature | OpenCode Equivalent |
|--------------------------|-------------------|
| LLM provider routing (OpenAI, Anthropic) | Native provider support (75+ providers) |
| Session ID generation (`crypto.randomUUID()`) | OpenCode session management |
| WebSocket streaming (`/workflows/stream`) | OpenCode real-time streaming |
| Checkpoint / state resumption (SQLite/Redis) | OpenCode session history |
| Constrained decoding / TypeBox validation | MCP tool schema validation + Zod in SDK |
| Schema degradation detection | SDK-level Zod validation on return |
| Financial circuit breaker | OpenCode cost tracking (Zen platform) |
| Browser automation (Playwright) | OpenCode built-in browser tools |
| Docker sandboxing | OpenCode permission system |
| API key timing-safe comparison | OpenCode auth layer |
| Graceful shutdown (SIGTERM drain) | OpenCode process management |
| Request body size limit (1MB) | OpenCode server hardening |
| HTTP security headers | OpenCode server hardening |

**Migration note:** The `openclaw-gateway/` directory is **archived** at `archived/openclaw-gateway/`. It is no longer part of the active build. The gateway code is preserved for historical reference and may be referenced for the TypeBox validation and schema degradation logic that is now embedded in the generated MCP server.

## 6. Security Model (OpenCode Context)

OpenCode handles the runtime security layer. Claw's compiler security invariants remain unchanged:

### 6.1 Compiler Security (unchanged from `specs/12`)
- Parser MUST NOT panic on any user input: use `Result<T, CompilerError>` everywhere
- `.expect()` only with `// SAFETY:` proof
- Numeric overflow ŌåÆ `CompilerError::ParseError` with span

### 6.2 MCP Server Security (new)
- All tool inputs are validated against JSON Schema before handler invocation
- `invoke: module(...)` paths are resolved relative to the workspace root
- Path traversal: the MCP server resolves module paths with `fs.realpath()` and verifies containment within the workspace root before dynamic `import()`
- MCP server is localhost-only (OpenCode starts it as a child process)
- No external network access for the MCP server process itself

### 6.3 OpenCode Handles (delegated)
- Session ID generation
- API key comparison
- WebSocket frame safety
- Request body size limits
- HTTP security headers
- Tool permission sandboxing

## 7. `env()` Expression Resolution

Unchanged from `specs/07 ┬¦5`:

- **Compile time:** `env("VAR")` is parsed as a function call expression and serialized into emitted config as OpenCode's `{env:VAR}` syntax.
- **Runtime (OpenCode):** OpenCode resolves `{env:VAR}` from the process environment when loading `opencode.json`.
- If the variable is not set, OpenCode fails at startup with a clear error.

## 8. Execution Flow: What Happens When a Developer Runs a Workflow

### 8.1 Interactive Use (via OpenCode CLI)

```bash
opencode /AnalyzeCompetitors "Apple, Microsoft, Google"
```

1. OpenCode loads `opencode.json` ŌåÆ reads top-level `model`, starts `mcp.claw-tools` subprocess, loads `instructions`
2. OpenCode loads `.opencode/command/AnalyzeCompetitors.md`, substitutes `$COMPANIES` with the user's argument
3. The `claw-tools` MCP server (`node generated/mcp-server.js`) is now running and ready
4. The coder agent executes, guided by `generated/claw-context.md`; it calls `agent_Researcher` and `agent_SeniorResearcher` as MCP tools
5. Each `agent_<Name>` MCP handler calls the LLM provider API **directly** (Anthropic SDK / Ollama), runs the tool-call loop internally, and returns the typed result ŌĆö no child `opencode` process is spawned
6. Output is streamed back to the terminal in real time

### 8.2 Programmatic Use (via TypeScript SDK)

**`--lang opencode`** generates OpenCode config files only ŌĆö no programmatic SDK.
Use **`--lang ts`** for programmatic access:

```bash
claw build --lang ts      # generates generated/claw/index.ts
```

```typescript
import { AnalyzeCompetitors } from "./generated/claw/index.js"

const reports = await AnalyzeCompetitors(["Apple", "Microsoft"])
// reports: string[] ŌĆö Zod-validated at the SDK boundary
```

The TypeScript SDK (`--lang ts`) is standalone and does not depend on OpenCode or any gateway. It directly orchestrates LLM calls using the provider APIs defined in `client` blocks. See `specs/06-CodeGen-SDK.md` for the SDK generation spec.

**Recommended workflow:** Use `--lang opencode` for interactive development and demos. Use `--lang ts` or `--lang python` for production application embedding.

### 8.3 Programmatic Use (via Python SDK)

```python
from generated.claw import analyze_competitors

reports = await analyze_competitors(["Apple", "Microsoft"])
# reports: List[str] ŌĆö Pydantic-validated
```

## 11. OpenCode Installation Requirement

Users MUST have OpenCode installed to run Claw workflows interactively:

```bash
# Install OpenCode (once)
curl -fsSL https://opencode.ai/install | bash

# Or via npm
npm install -g opencode
```

The `claw init` command detects whether OpenCode is installed and prints a warning if not found. The compiler itself (`clawc`) has no runtime dependency on OpenCode ŌĆö it only generates files.
