# Spec 34: Advanced Tool Patterns

**Status:** Specced 2026-03-19. Adds three Anthropic-derived efficiency patterns to the Claw DSL: Tool Registry with deferred loading (§2), Sandbox primitive with Tool Bridge (§3), and multi-shot Examples block (§4).

---

## 1. Motivation

The existing tool model loads every tool definition into the LLM context upfront. At scale this is expensive. Three patterns from Anthropic's advanced tool-calling research address this:

| Pattern | Problem solved | Token impact |
|---|---|---|
| Tool Registry + deferred loading | All tools loaded upfront even if unused | −50% on large tool sets |
| Sandbox + Tool Bridge | LLM makes N round-trips for iterative tasks | −80% for batch operations |
| Multi-shot Examples | Complex parameters hallucinated without grounding | +18% accuracy on complex params |

These patterns slot into the existing synthesis pipeline (Spec 32) without changing the core model: LLM writes TypeScript at build time, TypeScript runs deterministically.

## 2. Tool Registry with Deferred Loading

### 2.1 Concept

Instead of declaring `tools = [A, B, C]` and injecting all definitions into the agent's context, a `registry {}` block holds the tool set and the agent discovers tools at runtime via a generated `tool_search` MCP tool.

The agent's initial context contains only the `tool_search` tool definition (~500 tokens) rather than all N tool definitions (potentially 500×N tokens).

### 2.2 DSL Grammar

```
registry <Name> {
    tools  = [<ToolName>, ...]
    search = semantic | all
}
```

`search: semantic` — tools are loaded lazily. The agent calls `tool_search(query)` to discover relevant tools, then calls them. This is the deferred-loading pattern.

`search: all` — all tool definitions are injected at agent context time (current behavior for agents with `tools = [...]`). This is the backward-compatible default if a registry is referenced.

**Agent with registry reference:**

```claw
registry ToolRegistry {
    tools  = [WebSearch, Screenshot, DataQuery, Summarize, ParseDate]
    search = semantic
}

agent Researcher {
    client        = MyClaude
    tools         = ToolRegistry      // registry ref, not a list
    system_prompt = "Always call tool_search before calling any tool."
}
```

**Agent with explicit list (unchanged — backward compatible):**

```claw
agent SimpleAgent {
    client = MyClaude
    tools  = [WebSearch]              // still valid — all loaded upfront
}
```

### 2.3 Generated MCP tools (for semantic search)

When an agent references a registry with `search: semantic`, the MCP server (`generated/mcp-server.js`) gains two additional tools:

```typescript
// Auto-generated by claw compile when registry with search:semantic exists

tool_search(query: string) -> ToolMatch[]
// Searches the registry by embedding similarity.
// Returns tool names + brief descriptions + input schemas.
// Does NOT return the full implementation.

tool_load(name: string) -> ToolDefinition
// Returns the full definition of a tool so the agent can call it.
// The agent calls this before making the actual tool call.
```

The compiler generates both. `tool_search` uses a pre-built embedding index over the tool descriptions stored in the artifact. For offline/local deployments, it falls back to keyword search (BM25) with no embedding API required.

### 2.4 Artifact additions

```json
{
  "registries": [
    {
      "name":   "ToolRegistry",
      "search": "semantic",
      "tools":  ["WebSearch", "Screenshot", "DataQuery", "Summarize", "ParseDate"],
      "index":  [
        { "name": "WebSearch",  "description": "Searches the web for a query. Returns URL and snippet.", "input_summary": "query: string" },
        { "name": "Screenshot", "description": "Takes a screenshot of a URL using a browser.", "input_summary": "url: string" },
        ...
      ]
    }
  ]
}
```

The `index[]` entries are the text corpus used for search. Their `description` field is sourced from:
1. A `description:` property on the tool declaration (see §4.2), if present.
2. Otherwise: synthesized by the compiler from the tool's name, inputs, and output type.

### 2.5 Semantic validation rules

- A registry must list only declared tools.
- An agent may declare either `tools = [List]` or `tools = RegistryName` — not both.
- If an agent declares `tools = RegistryName` and the registry does not exist → `UndefinedRegistry` error.
- A tool may appear in multiple registries.
- A registry with `search: all` compiles to the same opencode output as an explicit tool list (no `tool_search` generated for that agent).

## 3. Sandbox Primitive with Tool Bridge

### 3.1 Concept

For tasks requiring many tool calls over a collection (e.g., "check expenses for all 50 employees"), the standard pattern causes O(N) LLM round-trips. The Sandbox pattern inverts this:

1. The LLM (synthesis pass) writes a **script** that iterates over the collection and calls tools via a local HTTP Tool Bridge.
2. The script runs once in an **isolated container** (no direct internet access, no credentials).
3. The Tool Bridge proxies container calls back to the real tool implementations on the host, injecting credentials.

This reduces LLM round-trips from O(N) to O(1) for batch operations.

### 3.2 DSL Grammar

```
sandbox <Name> {
    runtime      = gvisor | docker | subprocess
    network      = bridge_only | none
    bridge_tools = [<ToolName>, ...]
    timeout_ms   = <number>        // optional, default 30000
}
```

`runtime`:
- `gvisor` — hardened container (recommended for production). Requires `runsc` binary.
- `docker` — standard Docker container. Requires Docker daemon.
- `subprocess` — local subprocess, no container. Dev/testing only.

`network = bridge_only` — container can reach only the Tool Bridge endpoint (localhost:PORT). All other network calls fail.
`network = none` — no network access at all. Script uses only bridge tools.

`bridge_tools` — the host-side tools the sandbox may invoke via the bridge.

**Tool using sandbox:**

```claw
sandbox LocalSandbox {
    runtime      = gvisor
    network      = bridge_only
    bridge_tools = [ExpenseAPI, DataQuery]
    timeout_ms   = 60000
}

tool ProcessExpenses(employees: Employee[]) -> ExpenseReport {
    using: sandbox(LocalSandbox)
    test {
        input:  { employees: [] }
        expect: { total_spend: !empty, line_items: !empty }
    }
}
```

### 3.3 What the synthesis pass generates

For a `using: sandbox(...)` tool, the Synthesis Pass generates TWO artifacts:

**A. The tool wrapper** (`generated/tools/ProcessExpenses.ts`):
```typescript
// SYNTHESIZED — orchestrates sandbox container + bridge
import { runSandbox } from '../runtime/sandbox.js';
import type { Employee, ExpenseReport } from '../types.js';

export async function ProcessExpenses(inputs: { employees: Employee[] }): Promise<ExpenseReport> {
  return runSandbox('LocalSandbox', inputs, 'ProcessExpenses.script.js');
}
```

**B. The sandbox script** (`generated/tools/ProcessExpenses.script.js`):
```javascript
// SYNTHESIZED — runs inside the isolated container
// Communicates back to host via Tool Bridge at process.env.CLAW_BRIDGE_URL
const bridge = process.env.CLAW_BRIDGE_URL;

async function callBridgeTool(name, args) {
  const res = await fetch(`${bridge}/call/${name}`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(args),
  });
  return res.json();
}

const employees = JSON.parse(process.env.CLAW_INPUT).employees;
const lineItems = [];

for (const emp of employees) {
  const expense = await callBridgeTool('ExpenseAPI', { employee_id: emp.id });
  lineItems.push({ employee: emp.name, amount: expense.total });
}

const total = lineItems.reduce((sum, e) => sum + e.amount, 0);
process.stdout.write(JSON.stringify({ total_spend: total, line_items: lineItems }));
```

### 3.4 Tool Bridge server (auto-generated)

The compiler generates `generated/runtime/bridge-server.ts`:

```typescript
// generated/runtime/bridge-server.ts — auto-generated. Do not edit.
// Listens on a random localhost port. Proxies calls from sandbox to host tools.
// Never exposed to the internet.

import { createServer } from 'node:http';
import { ExpenseAPI } from '../tools/ExpenseAPI.js';
import { DataQuery } from '../tools/DataQuery.js';

const BRIDGE_TOOLS: Record<string, Function> = { ExpenseAPI, DataQuery };

export function startBridge(): Promise<string> {
  return new Promise((resolve) => {
    const server = createServer(async (req, res) => {
      const name = req.url!.split('/call/')[1];
      const body = await readBody(req);
      const args = JSON.parse(body);
      const fn = BRIDGE_TOOLS[name];
      if (!fn) { res.writeHead(404); res.end(JSON.stringify({ error: 'unknown tool' })); return; }
      try {
        const result = await fn(args);
        res.writeHead(200, { 'Content-Type': 'application/json' });
        res.end(JSON.stringify(result));
      } catch (err: any) {
        res.writeHead(500);
        res.end(JSON.stringify({ error: err.message }));
      }
    });
    server.listen(0, '127.0.0.1', () => {
      const port = (server.address() as any).port;
      resolve(`http://127.0.0.1:${port}`);
    });
  });
}
```

### 3.5 Runtime sandbox runner (auto-generated)

`generated/runtime/sandbox.ts`:

```typescript
export async function runSandbox<I, O>(sandboxName: string, inputs: I, scriptName: string): Promise<O> {
  const bridgeUrl = await startBridge();
  // ... spawns container with:
  //   CLAW_INPUT=JSON.stringify(inputs)
  //   CLAW_BRIDGE_URL=bridgeUrl
  //   network limited to bridge host:port only
  // ... reads stdout as JSON, returns as O
}
```

### 3.6 Artifact additions

```json
{
  "sandboxes": [
    {
      "name":         "LocalSandbox",
      "runtime":      "gvisor",
      "network":      "bridge_only",
      "bridge_tools": ["ExpenseAPI", "DataQuery"],
      "timeout_ms":   60000
    }
  ]
}
```

### 3.7 Semantic validation rules

- A sandbox must list only declared tools in `bridge_tools`.
- `using: sandbox(Name)` where Name is not declared → `UndefinedSandbox` error.
- A tool with `using: sandbox(...)` may not also declare `invoke:` (already enforced by existing `invoke:`/`using:` mutual exclusion).
- `runtime: gvisor` — compiler emits a warning (not error) if `runsc` not found on PATH at build time. The generated code still builds; failure is at execution time.
- `runtime: subprocess` is forbidden in non-development builds (enforced if `CLAW_ENV=production`).
- A sandbox with `network: none` may not declare `bridge_tools` (bridge requires network access to localhost). This is a compile error: `SandboxNetworkConflict`.

## 4. Multi-shot Examples Block

### 4.1 Concept

For tool parameters with complex structure, the LLM synthesis pass (and any LLM calling the tool at runtime) benefits from concrete input/output examples. The `examples {}` block bakes these into the artifact as few-shot demonstrations.

Unlike `test {}` which runs at build time to validate the synthesized implementation, `examples {}` is a **specification aid** — it goes into:
1. The `.clawa` artifact → Synthesis Pass prompt (improves synthesis quality)
2. The generated TypeScript JSDoc (improves any LLM calling the tool at runtime)
3. The MCP tool schema's `examples` field (improves agent accuracy)

### 4.2 DSL Grammar

```
tool <Name>(<inputs>) -> <OutputType> {
    using:   <capability>
    description: "<natural language description>"   // optional — used in registry search
    examples {
        { input: { <field>: <value>, ... }, output: { <field>: <value>, ... } }
        { input: { <field>: <value>, ... }, output: { <field>: <value>, ... } }
    }
    test {
        // unchanged — runs at build time
    }
}
```

Multiple `examples {}` blocks are NOT supported — one block with multiple `{ input, output }` entries.

**Example:**

```claw
tool ParseDate(text: string) -> Date {
    using:       baml("ParseDateLLM")
    description: "Parses natural language date expressions into structured Date objects."
    examples {
        { input: { text: "next Tuesday" },   output: { year: 2026, month: 3, day: 24 } }
        { input: { text: "end of quarter" }, output: { year: 2026, month: 3, day: 31 } }
        { input: { text: "Q1 2025" },        output: { year: 2025, month: 1, day: 1  } }
    }
    test {
        input:  { text: "March 15th" }
        expect: { year: !empty, month: !empty, day: !empty }
    }
}
```

### 4.3 Artifact additions

```json
{
  "tools": [
    {
      "name":        "ParseDate",
      "description": "Parses natural language date expressions into structured Date objects.",
      "inputs":      [{ "name": "text", "type": "string" }],
      "output_type": "Date",
      "using":       "baml(ParseDateLLM)",
      "examples": [
        { "input": { "text": "next Tuesday" },   "output": { "year": 2026, "month": 3, "day": 24 } },
        { "input": { "text": "end of quarter" }, "output": { "year": 2026, "month": 3, "day": 31 } },
        { "input": { "text": "Q1 2025" },        "output": { "year": 2025, "month": 1, "day": 1  } }
      ],
      "tests": [...]
    }
  ]
}
```

### 4.4 Generated TypeScript (JSDoc injection)

```typescript

 *

 * // input:  { text: "next Tuesday" }
 * // output: { year: 2026, month: 3, day: 24 }

```

### 4.5 MCP tool schema additions

When `description:` and `examples {}` are present, the MCP server injects them into the tool schema:

```json
{
  "name":        "ParseDate",
  "description": "Parses natural language date expressions into structured Date objects.\n\nExamples:\n  Input: {\"text\":\"next Tuesday\"} → Output: {\"year\":2026,\"month\":3,\"day\":24}",
  "inputSchema": { ... }
}
```

### 4.6 Synthesis pass prompt injection

The examples are injected into the synthesis prompt under `## FEW-SHOT EXAMPLES`:

```
## FEW-SHOT EXAMPLES
Study these input/output pairs before writing your implementation.
Your implementation MUST reproduce these exact outputs for these inputs.

Input:  { text: "next Tuesday" }
Output: { year: 2026, month: 3, day: 24 }

Input:  { text: "end of quarter" }
Output: { year: 2026, month: 3, day: 31 }
```

### 4.7 Semantic validation rules

- `examples {}` output fields must be a subset of the tool's declared output type fields. Unknown fields → `UnknownExampleField` error.
- `examples {}` input fields must match the tool's declared arguments. Missing required args → `MissingExampleInput` error.
- `examples {}` is valid on both synthesis-path tools (`using:`) and invoke-path tools (`invoke:`).
- A tool may declare at most 10 examples (compiler warning above 10: examples are injected into prompts; too many bloats context).
- `description:` without `examples {}` is valid (description alone improves registry search).

## 5. Tool `description:` Property

All three patterns benefit from a `description:` string property on tool declarations. It is a first-class tool property alongside `using:`, `test {}`, etc.

```
tool WebSearch(query: string) -> SearchResult {
    using:       fetch
    description: "Searches the web for a query string. Returns the top URL and a text snippet."
}
```

Used by:
- Registry semantic search index (§2.4) — primary use
- MCP tool schema `description` field (§4.5) — always emitted when present
- Synthesis Pass prompt as tool intent description
- JSDoc in generated TypeScript

If `description:` is absent and the tool is in a registry with `search: semantic`, the compiler auto-generates a description from `<name>(<inputs>) -> <output>` and emits a warning:
```
warning: tool WebSearch is in a semantic registry but has no description:
         auto-generated: "WebSearch(query: string) -> SearchResult"
         Add a description: property for better search accuracy.
```

## 7. New Error Codes

| Code | Name | Trigger |
|---|---|---|
| E-R01 | `UndefinedRegistry` | Agent references a registry that is not declared |
| E-R02 | `UndefinedToolInRegistry` | Registry lists a tool that is not declared |
| E-S01 | `UndefinedSandbox` | `using: sandbox(Name)` where Name is not declared |
| E-S02 | `UndefinedBridgeTool` | `bridge_tools` lists a tool that is not declared |
| E-S03 | `SandboxNetworkConflict` | `network: none` with non-empty `bridge_tools` |
| E-E01 | `UnknownExampleField` | Example output references field not in output type |
| E-E02 | `MissingExampleInput` | Example input missing a required tool argument |
| W-R01 | `RegistryToolNoDescription` | Tool in semantic registry has no `description:` |
| W-S01 | `SubprocessRuntimeInProduction` | `runtime: subprocess` with `CLAW_ENV=production` |
| W-E01 | `TooManyExamples` | Tool has > 10 examples |

## 8. GAN Audit Amendments (R1–R3)

The following changes were applied after 3 rounds of adversarial audit. See `34-GAN-Audit.md` for the full finding log.

### 8.1 `description:` as a formal tool property (B1-01)

`description:` is a first-class tool property parsed as `ToolProperty::Description(String)`. It appears in the tool body alongside `using:`, `invoke:`, `test {}`, `examples {}`. Maximum 500 characters (W-E02 warning above limit). Applies to both synthesis-path and invoke-path tools.

### 8.2 `AgentTools` enum in AST (B1-02)

`AgentDecl.tools: Vec<String>` becomes `AgentDecl.tools: AgentTools`:

```rust
pub enum AgentTools {
    List(Vec<String>),      // tools = [A, B, C]
    Registry(String),       // tools = RegistryName
}
```

Semantic analyzer resolves `Registry(name)` → validates registry exists + validates each tool in the registry → existing `UndefinedTool` errors apply per-tool.

### 8.3 `SandboxContext` in `SynthesisRequest` (B1-03)

Added to Spec 33 §2.1. When `using: sandbox(...)`, the synthesis request includes:

```typescript
interface SandboxContext {
  sandbox_name:   string;
  runtime:        'gvisor' | 'docker' | 'subprocess';
  network:        'bridge_only' | 'none';
  bridge_tools:   string[];
  bridge_url_env: 'CLAW_BRIDGE_URL';
  bridge_token_env: 'CLAW_BRIDGE_TOKEN';
  timeout_ms:     number;
  node_version:   string;
}
```

The synthesis pass generates the two-file pattern (wrapper + script) iff `sandbox_context` is present in the request.

### 8.4 BM25 replaces embedding search (B1-04)

`search: semantic` uses BM25 keyword scoring pre-built at compile time. No runtime embedding API required. Semantic embeddings are a future extension (post v1.0). The option is renamed internally but `search: semantic` remains the DSL keyword.

### 8.5 Node ≥ 18 + sandbox node version (B1-05)

New optional `node_version` field on `sandbox {}`:

```
sandbox LocalSandbox {
    runtime      = gvisor
    network      = bridge_only
    bridge_tools = [ExpenseAPI]
    node_version = "22"        // optional, default "22"
}
```

Sandbox scripts use `import fetch from 'node-fetch'` for maximum compatibility. Compiler emits warning if host Node < 18.

### 8.6 Examples output restricted to scalars (B1-06)

`examples {}` output values are scalar literals only: string, int, float, boolean, or arrays of scalars. Nested object literals are rejected at parse time. Validation rule `E-E03: NestedObjectInExample`.

### 8.7 Deferred MCP registration for semantic registries (B2-01)

When `search: semantic`, the MCP server registers ONLY `tool_search` and `tool_load` at startup. Registry tools are NOT pre-registered. `tool_load(name)` dynamically makes the tool available for the current session. This gives genuine deferred loading.

`tool_search` parameters:
```typescript
tool_search(query: string, max_results?: number) -> ToolMatch[]
// max_results: defaults to registry.max_results (default 5)
```

`tool_load` returns schema only — not implementation, not `using:` capability, not synthesis metadata:
```typescript
tool_load(name: string) -> { name: string, description: string, inputSchema: JSONSchema }
```

### 8.8 Registry configuration extended (B2-06, B3-06)

```
registry ToolRegistry {
    tools         = [...]
    search        = semantic
    max_results   = 5      // optional, default 5
    min_relevance = 0.1    // optional, default 0.1
}
```

BM25 scores are normalized to [0.0, 1.0]. Tools with score < `min_relevance` are excluded from results.

### 8.9 `bridge_tools` restricted to synthesis-path (B2-03)

New error `E-S04: BridgeToolNotSynthesized` if a `bridge_tools` entry uses `invoke:` only. Bridge imports from `generated/tools/` which only contains synthesized TypeScript.

### 8.10 `reason {}` × sandbox warning (B2-04)

New warning `W-S02: AgentUsedInReasonHasSandboxTool` — emitted when a `reason {}` block's agent has tools with `using: sandbox(...)` in its tool set. This is a warning not an error (the sandbox tool may not be invoked during reasoning).

### 8.11 Bridge server security (B3-01, B3-02)

- Bridge generates `crypto.randomBytes(32)` token at startup, injects as `CLAW_BRIDGE_TOKEN` into container.
- Every bridge request requires `Authorization: Bearer <token>`.
- `CLAW_INPUT` is base64-encoded by the host and decoded by the sandbox script.
- Bridge validates tool `args` against declared input schema (Zod) before forwarding.

### 8.12 Bridge server lifecycle (B3-05)

`runSandbox()` closes the bridge server in a `finally` block — always, even on timeout. Port is released immediately after each sandbox invocation.

### 8.13 MCP example injection cap (B3-04)

MCP schema description: first 3 examples max, 80 chars per example, 500 char hard cap for examples section. All examples still appear in artifact + synthesis prompt.

## 9. What This Spec Does NOT Cover

- Embedding API choice or embedding model for semantic search (implementation detail of the generated `tool_search` tool)
- Container image used for gvisor/docker runtime (user provides or defaults to `node:22-slim`)
- `claw verify` sandbox E2E test execution (test tier 3 — see Spec 32 §7)
- Streaming responses from sandbox scripts (future: `using: sandbox` with `stream: true`)
- Multi-registry agents (an agent referencing more than one registry — future extension)
