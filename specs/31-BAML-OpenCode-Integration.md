# Spec 31: BAML Integration — OpenCode Architecture

**Status:** Replaces spec/18 (archived — described gateway architecture). Implementation complete as of 2026-03-19 (GAN audit verified all three wiring points).

**Honest scope:** BAML adds value only for tools that internally call an LLM. It does NOT apply to the main agent loop (OpenCode controls that), workflow orchestration (Claw controls that), or plain I/O tools. This is a tool-level concern only.

---

## 1. The Problem BAML Solves

OpenCode + Ollama/Claude makes the main LLM calls. Claw controls the workflow. But when a **tool itself needs to call an LLM** to produce a typed result, we have no structured output guarantee:

```
OpenCode calls MCP tool "ExtractKeywords"
→ tool makes raw LLM HTTP call
→ LLM returns "Here are the keywords: foo, bar" (not JSON)
→ MCP handler crashes trying to parse
→ OpenCode gets an error
```

BAML solves this: it wraps the inner LLM call, enforces the output schema, and retries if the LLM produces malformed output.

---

## 2. Where BAML Does NOT Apply

**Do not attempt BAML integration for:**

- The main agent loop (e.g., `agent Researcher` executing via OpenCode) — OpenCode owns those LLM calls, we can't inject BAML
- Workflow orchestration — Claw's control flow is already deterministic code
- Plain I/O tools — `WebSearch` via HTTP, DB queries, file reads don't call LLMs
- The MCP server runner for `agent_<Name>` tools — those spawn `opencode` subprocesses

---

## 3. New DSL Syntax: `invoke: baml(...)`

The only change to the `.claw` language is a new `invoke` target type:

```claw
// Existing: delegates to a JS module
tool WebSearch(query: string) -> SearchResult {
    invoke: module("scripts/search").function("run")
}

// NEW: delegates to a BAML function (LLM-powered, typed output)
tool ExtractKeywords(text: string) -> KeywordList {
    invoke: baml("ExtractKeywords")
}

tool ClassifyIntent(message: string) -> Intent {
    invoke: baml("ClassifyIntent")
}
```

The developer writes only `.claw`. BAML is invisible — `clawc` emits both the BAML definition and the MCP handler.

---

## 4. What Compiles to What

For a `baml(...)` tool, `clawc build` emits **two things** instead of one:

### 4.1 `generated/baml_src/functions.baml`

```baml
// AUTO-GENERATED by clawc build
// DO NOT EDIT — re-run clawc to regenerate

function ExtractKeywords(text: string) -> KeywordList {
  client LocalQwen
  prompt #"
    Extract keywords from the following text.
    Return a KeywordList with a list of keyword strings.

    Text: {{ text }}

    {{ ctx.output_format }}
  "#
}

function ClassifyIntent(message: string) -> Intent {
  client LocalQwen
  prompt #"
    Classify the intent of the following message.

    Message: {{ message }}

    {{ ctx.output_format }}
  "#
}
```

### 4.2 `generated/baml_src/types.baml`

```baml
// AUTO-GENERATED

class KeywordList {
  keywords string[]
}

class Intent {
  category string
  confidence float
}
```

### 4.3 `generated/baml_src/clients.baml`

```baml
// AUTO-GENERATED

client<llm> LocalQwen {
  provider openai-generic
  options {
    base_url "http://localhost:11434/v1"
    model "qwen2.5-coder:7b"
    api_key "ollama"
  }
}
```

### 4.4 MCP Handler for BAML Tool (in `generated/mcp-server.js`)

```js
// For baml-invoke tools, the handler calls the BAML client
async function handleExtractKeywords(args) {
  try {
    const { text } = args;
    const { b } = await import("../baml_client/index.js");
    const result = await b.ExtractKeywords({ text });
    validateOutput(result, SCHEMAS.KeywordList, "ExtractKeywords");
    return { content: [{ type: "text", text: JSON.stringify(result) }] };
  } catch (err) {
    return { content: [{ type: "text", text: `Error: ${err.message}` }], isError: true };
  }
}
```

## 5. Build Flow

```bash
# Step 1: Claw compiles .claw → MCP server + BAML source files
claw build

# Step 2: BAML CLI generates typed client from BAML source
npx @boundaryml/baml-cli generate --from generated/baml_src

# Result:
generated/
  mcp-server.js          ← MCP server (orchestration)
  baml_src/              ← BAML source definitions
    functions.baml
    types.baml
    clients.baml
  baml_client/           ← BAML generated typed client (from step 2)
    index.js
    types.js
```

`claw build` can run step 2 automatically if `@boundaryml/baml-cli` is installed:

```rust
// In src/bin/claw.rs, after codegen:
if baml_tools_exist && baml_cli_installed {
    run_baml_generate();
} else if baml_tools_exist {
    println!("⚠  BAML tools detected. Run: npx @boundaryml/baml-cli generate --from generated/baml_src");
}
```

## 6. When to Use BAML vs Module

| Use `invoke: module(...)` when | Use `invoke: baml(...)` when |
|---|---|
| Tool calls an HTTP API | Tool needs LLM to extract/classify/transform |
| Tool reads a file or database | Output schema must be guaranteed (no raw JSON) |
| Tool runs a shell command | LLM output needs automatic retry on malformed response |
| Tool is deterministic | The "processing" is itself a prompt |

---

## 7. What Existing `baml.rs` Needs

The existing `src/codegen/baml.rs` (710 lines) was built for the gateway architecture — it generates BAML functions for **agents**. This needs to be repurposed for **tools with `invoke: baml(...)` targets**.

### Changes needed:

1. **Remove**: `collect_call_sites()` and `resolve_agents()` — these were for agent-centric BAML generation
2. **Add**: `collect_baml_tools(document) -> Vec<&ToolDecl>` — filter tools where `invoke_path` starts with `baml(`
3. **Rewrite**: `emit_functions()` — generate one BAML function per baml-invoke tool (not per agent+return_type pair)
4. **Keep**: `emit_types()`, `emit_clients()`, `emit_generators()` — these are still correct
5. **Update**: `generate_baml()` public API to take `document` only, derive everything internally

The BAML client path reference in MCP handlers assumes `../baml_client/index.js` — the MCP server at `generated/mcp-server.js` and BAML client at `generated/baml_client/`.

---

## 8. Parser Change: `invoke: baml(...)`

The parser currently handles:
```
invoke: module("scripts/search").function("run")
```

Add support for:
```
invoke: baml("FunctionName")
```

In the AST, `ToolDecl.invoke_path` already stores this as a string. The distinction is:
- `invoke_path.starts_with("module(")` → JS module handler
- `invoke_path.starts_with("baml(")` → BAML handler

No AST changes needed — just parser and codegen.

---

## 9. BAML is Optional

Tools with `invoke: module(...)` continue to work with no BAML dependency. BAML is only activated when at least one tool uses `invoke: baml(...)`. `claw build` emits no BAML files if no BAML tools exist.

```
claw build (no baml tools) → mcp-server.js only
claw build (1+ baml tools) → mcp-server.js + baml_src/ + warning to run baml-cli generate
```

## 10. Success Criteria

```bash
grep "baml_client" generated/mcp-server.js  # finds the import

# End-to-end: OpenCode calls ExtractKeywords tool
# → MCP handler imports baml_client
# → BAML calls Ollama with structured prompt

# → validateOutput passes
# → OpenCode receives guaranteed-typed response
```

---

## 11. What This Does NOT Do

- Does not make agent output (from `execute Researcher.run(...)`) go through BAML — OpenCode owns those calls
- Does not add BAML to workflow control flow — Claw's `if`, `for`, `let` remain as-is
- Does not require BAML for any existing functionality — fully backwards-compatible
- Does not replace `validateOutput()` in non-BAML handlers — BAML is additive

---

## 12. Implementation — Three Required Wiring Points

BAML integration spans three files that must ALL be updated together. Missing any one silently breaks the feature.

### 12.1 `src/codegen/baml.rs` — generates BAML source content

`generate_baml(document)` returns `BamlOutput { generators, clients, types, functions }`.

The client name in emitted BAML functions MUST come from `document.clients.first().name`, not the hardcoded string `"DefaultClient"`. If you hardcode `"DefaultClient"` and the user's `.claw` file names their client `LocalQwen`, BAML will fail to compile with a missing client reference error.

```rust
let default_client = document.clients.first().map(|c| c.name.as_str()).unwrap_or("DefaultClient");
// pass default_client through to emit_functions() and emit_function()
```

### 12.2 `src/bin/claw.rs` — build pipeline must call `generate_baml()`

In `run_compile_once()`, the `BuildLanguage::Opencode` branch must call `generate_baml()` and write the output files. `generate_baml` is exported from `codegen` but if not called, no BAML files are ever written — the user gets no error, just silently missing output.

```rust
let baml_output = codegen::generate_baml(&document)?;
if !baml_output.functions.is_empty() {
    let baml_dir = project_root.join("generated").join("baml_src");
    fs::create_dir_all(&baml_dir)?;
    fs::write(baml_dir.join("generators.baml"), baml_output.generators)?;
    fs::write(baml_dir.join("clients.baml"), baml_output.clients)?;
    fs::write(baml_dir.join("types.baml"), baml_output.types)?;
    fs::write(baml_dir.join("functions.baml"), baml_output.functions)?;
    println!("  BAML tools detected. Run: npx @boundaryml/baml-cli generate --from generated/baml_src");
}
```

### 12.3 `src/codegen/mcp.rs` — `emit_handler()` must branch on `baml(...)`

`emit_handler()` must check `invoke_path.starts_with("baml(")` before attempting to parse it as `module(...)`. Without this check, a `baml(...)` tool falls through to the module handler, which dot-splits `"baml(\"ExtractKeywords\")"` into a single-element array, fails the `module("` check, and emits a handler that tries to import `../scripts/stub.js` — runtime error at the point OpenCode calls the tool.

The BAML handler must import from `../baml_client/index.js` (the BAML-CLI-generated typed client):

```js
async function handleExtractKeywords(args) {
  const { text } = args;
  const { b } = await import("../baml_client/index.js");
  const result = await b.ExtractKeywords({ text });
  // ... validateOutput + return
}
```

Note: `../baml_client/` is relative to `generated/mcp-server.js`, so the actual path is `generated/baml_client/index.js` from the project root.
