# Spec 38: Claw Runtime — Headless Workflow Execution

**Status:** Specced 2026-03-20. Revised from initial draft after GAN audit and architecture clarification.
**Depends on:** Spec 25 (OpenCode Integration), Spec 26 (MCP Server Generation)
**Referenced by future specs:** Spec 39 (Control Flow Compilation — planned, not yet written)

---

## 1. The Use Case This Spec Solves

The current Claw build output is designed for **interactive use** in the OpenCode IDE. A developer types `/Summarize topic="quantum computing"` in a chat session and gets a response. That is the right model for development and exploration.

But Claw workflows are also useful outside the IDE:

- **CI/CD pipelines** — analyze a PR, generate release notes, run a threat model on a diff, score test coverage, produce a structured deploy report
- **Scheduled jobs** — daily research digests, weekly competitive analysis, nightly data summaries
- **Service integration** — embed a typed, LLM-powered workflow step into a larger system via HTTP

None of these use cases have a human at a chat window. They need:

1. A single command that runs a workflow and exits
2. Typed JSON on stdout (parseable by the next pipeline step)
3. A meaningful exit code (0 = success, non-zero = failure)
4. No OpenCode process running

This spec adds that execution path. **OpenCode integration is preserved unchanged.** `claw run` is additive — a second way to execute workflows alongside the existing IDE path.

## 2. Architecture

### 2.1 What the build produces today

```text
claw build demo.claw
  → opencode.json               ← OpenCode config (model, mcp, instructions)
  → .opencode/command/*.md      ← workflow command stubs for IDE slash commands
  → generated/mcp-server.js    ← MCP server: tools + agent runners (call LLM directly)
  → generated/claw-context.md  ← project context document
```

### 2.2 What this spec adds

```text
claw build demo.claw
  → [all of the above, unchanged]
  → generated/runtime.js        ← NEW: headless workflow runner
```

### 2.3 Execution paths side by side

```text
INTERACTIVE (OpenCode IDE — unchanged)
  User types /Summarize topic="quantum computing" in chat
  → OpenCode reads .opencode/command/Summarize.md
  → OpenCode coder LLM calls agent_Writer MCP tool
  → mcp-server.js: handleagent_Writer calls Ollama directly, tool-call loop
  → typed result returned to OpenCode, streamed to chat UI

HEADLESS (CI/CD, scripts, services — NEW)
  claw run Summarize --arg topic="quantum computing"
  → Rust binary spawns: node generated/runtime.js Summarize --arg topic=...
  → runtime.js executes workflow plan deterministically
  → calls agent_Writer loop directly (no OpenCode, no MCP)
  → validates result against Summary schema
  → stdout: {"title":"...","body":"...","confidence":0.91}
  → exit 0
```

### 2.4 Why the agent loop is the same in both paths

The `handleagent_Writer` function in `mcp-server.js` already calls the LLM provider API directly (Spec 26 §2.5). The `runtime.js` agent loop is the same code — same Anthropic SDK call, same Ollama fetch, same tool dispatch. The difference is the entry point and output format:

| | `mcp-server.js` | `generated/runtime.js` |
| --- | --- | --- |
| Entry | MCP `tools/call` request | CLI arg `claw run Workflow` |
| Exit | MCP response JSON | stdout + process exit code |
| Lifetime | Long-running process | Single execution, exits |
| Caller | OpenCode (via MCP protocol) | Rust `claw` binary (via `std::process::Command`) |
| Agent loop | `handleagent_<Name>` | `runAgent("<Name>", task, scope)` |

Both share the same underlying LLM call pattern. The codegen emits them from the same Rust functions.

## 3. `generated/runtime.js` — The Headless Runner

Generated alongside `mcp-server.js` by `claw build --lang opencode`. Self-contained ESM file.

### 3.1 Structure

```javascript
// generated/runtime.js
// AUTO-GENERATED by clawc build --lang opencode
// DO NOT EDIT — re-run clawc to regenerate

// ── Workflow Plans ────────────────────────────────────────────────────────────
// Compiled deterministic execution plans — one per workflow block

const PLANS = {
  Summarize: {
    name: "Summarize",
    requiredArgs: ["topic"],
    returnType: "Summary",
    steps: [
      {
        type: "execute_agent",
        bind: "result",
        agent: "Writer",
        taskTemplate: "Write a concise summary about: ${topic}",
        requireType: "Summary",
      },
      {
        type: "return",
        value: { type: "var", name: "result" },
      },
    ],
  },
  // ... one entry per workflow block
};

// ── Type Schemas (for output validation) ─────────────────────────────────────

const SCHEMAS = {
  Summary: {
    type: "object",
    properties: {
      title: { type: "string" },
      body: { type: "string" },
      confidence: { type: "number" },
    },
    required: ["title", "body", "confidence"],
  },
  // ... one entry per type block
};

// ── Agent Configs ─────────────────────────────────────────────────────────────

const AGENTS = {
  Writer: {
    provider: "local",
    model: "qwen2.5:14b",
    systemPrompt: "You are a concise technical writer...",
    tools: ["WebSearch"],
    maxSteps: 3,
    temperature: 0.2,
  },
  // ... one entry per agent block
};

// ── Tool Handlers ─────────────────────────────────────────────────────────────
// Same implementation as mcp-server.js handlers

async function callTool(name, input) { /* ... identical to MCP dispatch */ }

// ── Agent Runner ──────────────────────────────────────────────────────────────

async function runAgent(agentName, task) {
  const agent = AGENTS[agentName];
  // provider-specific LLM loop (Anthropic SDK or Ollama fetch)
  // identical logic to handleagent_<Name> in mcp-server.js
  // returns the agent's text output
}

// ── Workflow Executor ─────────────────────────────────────────────────────────

async function executeWorkflow(planName, args) {
  const plan = PLANS[planName];
  if (!plan) throw { code: "E-RUN01", message: `workflow "${planName}" not found` };

  for (const required of plan.requiredArgs) {
    if (!(required in args)) throw { code: "E-RUN02", message: `missing argument "${required}"` };
  }

  const scope = { ...args };

  for (const step of plan.steps) {
    if (step.type === "execute_agent") {
      const task = interpolate(step.taskTemplate, scope);
      const output = await runAgent(step.agent, task);
      if (step.requireType) {
        scope[step.bind] = parseAndValidate(output, step.requireType);
      } else {
        scope[step.bind] = output;
      }
    } else if (step.type === "return") {
      return resolveValue(step.value, scope);
    }
  }
}

// ── CLI Entry ─────────────────────────────────────────────────────────────────

const [,, workflowName, ...rawArgs] = process.argv;
const args = parseArgs(rawArgs);  // "--arg key=value" → { key: "value" }

try {
  const result = await executeWorkflow(workflowName, args);
  process.stdout.write(JSON.stringify(result) + "\n");
  process.exit(0);
} catch (err) {
  process.stderr.write(JSON.stringify({ error: err.message, code: err.code ?? "E-RUN99" }) + "\n");
  process.exit(exitCodeFor(err.code));
}
```

### 3.2 Workflow plan compilation

Each `workflow` block compiles to a plan entry in `PLANS`. The plan encodes the workflow body as a sequence of typed steps — no natural language, no LLM interpretation.

**Supported step types in v1:**

| Step type | Compiled from | Notes |
| --- | --- | --- |
| `execute_agent` | `execute Agent.run(task: "...", require_type: T)` | Most common |
| `return` | `return <expr>` | Final step |
| `let` | `let x: T = <literal_expr>` | Non-agent assignments |

**For v1, complex control flow (`for`, `if`, nested agents) compiles to `execute_agent` steps that describe the intent.** Full control flow compilation is deferred to a future spec (planned as Spec 39, not yet written).

### 3.3 Argument parsing

`--arg key=value` pairs. Multiple `--arg` flags allowed. Values are always strings at the CLI boundary; the runtime coerces to the declared type.

```bash
claw run Summarize --arg topic="quantum computing"
claw run AnalyzePR --arg pr_number=42 --arg repo=myorg/myrepo
```

## 4. `claw run` — CLI Command

```bash
claw run <workflow> [--arg <key>=<value>]... [--output <file>] [--timeout <seconds>] [--client <name>]
```

### 4.1 Behaviour

1. Locates `generated/runtime.js` by walking up from CWD to find the nearest `claw.json`.
2. Resolves the Node.js binary (same `find_node_binary()` logic as `mcp-server.js`).
3. Spawns: `node generated/runtime.js <workflow> --arg key=value ...`
4. Pipes stdout to its own stdout (or `--output <file>`).
5. Pipes stderr to its own stderr.
6. Exits with the same exit code as the node process.

### 4.2 Exit codes

| Code | Meaning | When |
| --- | --- | --- |
| `0` | Success | Workflow completed, result written to stdout |
| `1` | Workflow not found | `<workflow>` does not match any compiled plan |
| `2` | Missing argument | Required workflow argument not provided |
| `3` | Type validation failed | LLM output did not match declared return type |
| `4` | LLM / network error | Provider API call failed (timeout, auth, etc.) |
| `5` | Config error | No `generated/runtime.js` found; not built yet |

Exit code semantics follow Unix convention: 0 = success, non-zero = failure. CI systems use these to fail pipeline steps automatically.

### 4.3 Type signature

```rust
#[derive(Debug, clap::Args)]
struct RunArgs {
    /// Workflow name to execute
    workflow: String,
    /// Arguments as key=value pairs (repeatable)
    #[arg(long = "arg", value_name = "KEY=VALUE")]
    args: Vec<String>,
    /// Write result JSON to file instead of stdout
    #[arg(long)]
    output: Option<PathBuf>,
    /// Timeout in seconds (default: 120)
    #[arg(long, default_value = "120")]
    timeout: u64,
    /// Override the client for all agents (default: each agent uses its compiled client from .claw).
    /// When absent, runtime.js uses the provider/model baked in at build time.
    /// Pass a client name to override: e.g. --client MyClaude.
    #[arg(long)]
    client: Option<String>,
}
```

**`--client` default:** When `--client` is not passed, `runtime.js` uses the provider and model that were compiled into it from the `.claw` file (each agent uses its own declared `client`). `--client <name>` overrides every agent in the workflow to use the named client, which must exist in `claw.json`. If the name is not found in `claw.json`, `claw run` exits with E-RUN05.

## 5. `claw serve` — HTTP Wrapper

```bash
claw serve [--port <n>]
```

Starts a local HTTP server. Each workflow is a POST endpoint. Useful for embedding Claw workflows in larger services.

```http
POST /run/Summarize
Content-Type: application/json
{ "topic": "quantum computing" }

→ 200 OK
{ "title": "...", "body": "...", "confidence": 0.91 }

→ 400 Bad Request (missing argument)
{ "error": "missing argument \"topic\"", "code": "E-RUN02" }

→ 422 Unprocessable Entity (type validation failed)
{ "error": "LLM output did not match Summary schema", "code": "E-RUN03" }

→ 503 Service Unavailable (LLM unavailable)
{ "error": "Ollama not reachable at http://localhost:11434", "code": "E-RUN04" }
```

`claw serve` is implemented as a thin Rust HTTP server that calls `node generated/runtime.js` per request and converts the exit code + output to an HTTP response.

```rust
#[derive(Debug, clap::Args)]
struct ServeArgs {
    #[arg(long, default_value = "3000")]
    port: u16,
    #[arg(long)]
    client: Option<String>,
}
```

## 6. CI/CD Integration

### 6.1 GitHub Actions

```yaml
# .github/workflows/analyze.yml
jobs:
  claw-analysis:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install Claw
        run: curl -fsSL https://claw.dev/install | bash

      - name: Build Claw workflows
        run: claw build workflows.claw

      - name: Analyze PR
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claw run AnalyzePR \
            --arg pr_number=${{ github.event.number }} \
            --arg diff="$(git diff HEAD~1 --stat)" \
            > pr-analysis.json

      - name: Post analysis as comment
        uses: actions/github-script@v7
        with:
          script: |
            const analysis = require('./pr-analysis.json');
            github.rest.issues.createComment({
              issue_number: ${{ github.event.number }},
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## Analysis\n${analysis.summary}`
            });
```

### 6.2 GitLab CI

```yaml
# .gitlab-ci.yml
threat-model:
  stage: security
  image: node:20
  before_script:
    - curl -fsSL https://claw.dev/install | bash
    - claw build security.claw
  script:
    - claw run ThreatModel --arg diff="$(git diff HEAD~1)" > threat-report.json
    - |
      SEVERITY=$(cat threat-report.json | node -e "const d=require('fs').readFileSync('/dev/stdin','utf8'); console.log(JSON.parse(d).severity)")
      if [ "$SEVERITY" = "critical" ]; then exit 1; fi
  artifacts:
    paths:
      - threat-report.json
```

### 6.3 Why this is powerful

The combination of **declared types + LLM intelligence + deterministic artifact** means:

- The pipeline gets a **typed** result, not an unstructured blob
- The LLM reasoning happens inside the workflow — the pipeline step is just `claw run`
- **Failure is explicit** — type validation means the pipeline knows if the LLM didn't produce valid output
- Results are **composable** — one `claw run` step's output can be piped to another's `--arg`

## 7. Codegen Changes

### 7.1 New file

`src/codegen/runtime.rs` — generates `generated/runtime.js`.

```rust
pub fn generate(document: &Document, project_root: &Path) -> CompilerResult<()>
```

Key sub-functions:

```rust
fn emit_plans(document: &Document) -> String
// Emits PLANS object: one entry per workflow, steps array

fn emit_agents(document: &Document) -> String
// Emits AGENTS object: provider, model, systemPrompt, tools, maxSteps, temperature

fn emit_agent_runner(document: &Document) -> String
// Emits runAgent() — identical LLM call logic to mcp.rs emit_agent_handler
// For Anthropic: uses @anthropic-ai/sdk
// For local: uses fetch to http://localhost:11434/v1/chat/completions

fn emit_tool_handlers(document: &Document) -> String
// Emits callTool() + individual tool handler functions
// Identical to mcp.rs emit_handler output
```

**Shared logic between `mcp.rs` and `runtime.rs`:** Both emit the same LLM call loop and tool handlers. To avoid duplication, a private `emit_llm_loop_anthropic(model, system_prompt, max_steps, temperature, tools_filter)` helper in `src/codegen/shared_js.rs` (new file) is used by both.

### 7.2 Existing files changed

| File | Change |
| --- | --- |
| `src/codegen/mod.rs` | Add `pub mod runtime;` and `pub mod shared_js;`; call `runtime::generate()` from `generate_opencode()` |
| `src/bin/claw.rs` | Add `Run(RunArgs)` and `Serve(ServeArgs)` to `Commands` enum; implement handlers |
| `src/codegen/mcp.rs` | `emit_agent_handler_anthropic` and `emit_agent_handler_ollama` moved to `shared_js.rs`, called from `mcp.rs` |

### 7.3 No AST changes

This spec requires no new DSL syntax and no new AST nodes. `generated/runtime.js` is always generated when `--lang opencode` is used (same condition as `mcp-server.js`). A `runtime {}` DSL block for controlling outputs is deferred to a future spec.

## 8. Error and Warning Codes

| Code | Name | Trigger |
| --- | --- | --- |
| E-RUN01 | WorkflowNotFound | `claw run <name>` where `<name>` not in compiled PLANS |
| E-RUN02 | ArgMissing | Required workflow argument not in `--arg` flags |
| E-RUN03 | TypeValidationFailed | LLM output failed schema validation for return type |
| E-RUN04 | ProviderUnreachable | LLM API call failed (network error, auth failure, timeout) |
| E-RUN05 | RuntimeNotBuilt | `generated/runtime.js` not found — run `claw build` first |
| E-RUN06 | ServePortInUse | `claw serve --port N` — port already bound |
| W-RUN01 | ComplexFlowTruncated | Workflow has `for`/`if` blocks compiled to simplified steps in v1 |

### 8.1 Error examples

```bash
# E-RUN01
$ claw run DoesNotExist
error[E-RUN01]: workflow "DoesNotExist" not found
  available: Summarize, AnalyzePR, GenerateChangelog
  hint: run `claw build` to recompile if you added a new workflow

# E-RUN02
$ claw run Summarize
error[E-RUN02]: missing required argument "topic" for workflow "Summarize"
  usage: claw run Summarize --arg topic=<value>

# E-RUN03
$ claw run Summarize --arg topic="test"
error[E-RUN03]: type validation failed — return type "Summary" not satisfied
  expected fields: title (string), body (string), confidence (number)
  got: {"title":"test","summary":"..."}  ← "summary" is not "body"

# E-RUN04
$ claw run Summarize --arg topic="test"
error[E-RUN04]: provider unreachable
  Ollama not responding at http://localhost:11434
  hint: start Ollama with `ollama serve` or set ANTHROPIC_API_KEY for cloud

# E-RUN05
$ claw run Summarize --arg topic="test"
error[E-RUN05]: runtime not built — generated/runtime.js not found
  run `claw build demo.claw` first

# E-RUN06
$ claw serve --port 3000
error[E-RUN06]: port 3000 already in use
  hint: use --port <n> to choose a different port, or stop the process on port 3000

# W-RUN01 (emitted at build time, not runtime)
$ claw build complex.claw
warning[W-RUN01]: workflow "ProcessBatch" contains `for` loop — compiled to single execute_agent step in v1
  full loop semantics will be available in Spec 39 (control flow compilation)
  the generated step describes the loop intent to the agent as task context
```

## 9. Edge Cases

### 9.1 Workflow with no return type

If `workflow Foo(x: string)` has no `-> T`, the runner outputs `null` as JSON and exits 0. No type validation runs.

### 9.2 Workflow with complex control flow (`for`, `if`)

In v1, `for` and `if` steps are compiled as a single `execute_agent` step with the loop/condition described as task context. W-RUN01 is emitted at build time. Full control flow compilation is Spec 39.

### 9.3 `claw run` called with no `generated/runtime.js`

Exits with E-RUN05, exit code 5. Does NOT attempt to auto-build — auto-build would silently call the LLM during what should be a read-only check step in CI.

### 9.4 `--timeout` exceeded

If the node process exceeds `--timeout` seconds, the Rust `claw run` binary sends SIGTERM. `runtime.js` exits with code 4. Exit code propagated.

### 9.5 `claw serve` + multiple concurrent requests

Each request spawns its own `node generated/runtime.js` process. No shared state between requests. Concurrency is bounded by available system resources. Rate limiting is the caller's responsibility.

### 9.6 Conflict: `claw run` and OpenCode running simultaneously

Both can run at the same time. `claw run` spawns a fresh node process independent of any running `opencode` or `mcp-server.js` instance. No port conflicts, no shared files.

## 10. Offline Behaviour

`claw run` checks at startup before spawning node:

1. If no `generated/runtime.js` exists → exit with E-RUN05 immediately (no node spawn).
2. If `ANTHROPIC_API_KEY` not set AND no Ollama running → node process will fail with E-RUN04. The Rust wrapper does not pre-check the provider (avoids false negatives from transient network).
3. `--timeout 0` disables the timeout (for long-running research workflows in local environments).

For local Ollama workflows in CI: set `OLLAMA_HOST` environment variable to override the default `http://localhost:11434`. The generated `runtime.js` respects this env var.

## 11. What This Spec Does NOT Cover

- Full control flow compilation (`for`, `if`, nested workflows) — Spec 39
- `runtime {}` DSL block for controlling which outputs are generated — future spec
- Docker base image for Claw CI — distribution concern, not compiler concern
- Official GitHub Action (`uses: claw-lang/run-action@v1`) — future tooling spec
- Streaming output from `claw run` (current: blocks until workflow completes) — future spec
- Budget caps per `claw run` invocation — future spec (builds on Spec 36 budget tracking)
