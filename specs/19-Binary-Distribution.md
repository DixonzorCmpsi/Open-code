# Phase 7: Binary Distribution & Portable Installation

## 1. Goal & Rationale
Currently, installing the `claw` compiler requires a local Rust toolchain (`cargo install --path .`), which is a significant barrier to entry for standard Node.js/Python developers. 

The objective of this phase is to convert the `claw` language installation into a highly portable, zero-dependency installation via NPM and standalone Shell scripts—mirroring the world-class Developer Experience (DX) of tools like BAML, Prisma, and TailwindCSS.

---

## 2. GitHub Actions Matrix (CI/CD)

The core mechanism for distribution is a cross-compiled build matrix in `.github/workflows/release.yml`.

### Triggers
- Automatic execution upon pushing a SemVer tag (e.g., `v1.0.0`).

### Target Matrix
The workflow will compile the raw Rust `claw` binary across all primary operating systems and CPU architectures:
- `x86_64-unknown-linux-gnu` (Ubuntu Linux)
- `aarch64-unknown-linux-gnu` (Linux ARM64 / AWS Graviton)
- `x86_64-apple-darwin` (macOS Intel)
- `aarch64-apple-darwin` (macOS Apple Silicon / M-series)
- `x86_64-pc-windows-msvc` (Windows x64)

### Artifact Publishing
Each compiled binary will be compressed (`.tar.gz` for UNIX, `.zip` for Windows) and published securely as Assets to a GitHub Release synchronized exactly with the git tag.

---

## 3. The NPM Wrapper (`@claw/cli`)

To capture the vast TypeScript/JavaScript market, we will publish a tiny metadata-only NPM package named `@claw/cli`. 
This package contains ZERO Rust code—only a `package.json` and a `postinstall.js` script.

### Installation Flow
1. Developer runs `npm install -g @claw/cli` (or saves it as a `devDependency`).
2. NPM executes the `postinstall` script phase.
3. The `postinstall.js` script queries the local machine for its OS (`os.platform()`) and CPU (`os.arch()`).
4. It maps the local machine to the corresponding GitHub Release URL (e.g., downloading `claw-aarch64-apple-darwin.tar.gz` for a current-gen MacBook).
5. The script downloads the archive, unpacks it into the package's local `bin/` directory, and registers the binary directly to the system's `$PATH`.

**Result:** The developer immediately has access to the `claw` command natively without ever touching Cargo.

## 4. Downstream Architectural Impacts & Dependencies

Transitioning to pre-compiled binaries fundamentally impacts how the compiler interacts with the rest of the workspace.

### 4.1 CLI Tooling (`specs/14-CLI-Tooling.md`)
- **Impact**: `claw init` must be updated to scaffold a standard `package.json` that includes `@claw/cli` as a `devDependency` instead of expecting a global source installation.
- Developers will now run compiler commands heavily through `$ npx claw build` or `$ npm run dev` instead of raw `claw`.

### 4.2 Gateway Version Synchronization (`specs/07-OpenClaw-OS.md`)
- **Impact**: The TypeScript execution Gateway must perfectly match the version of the raw AST generated by the Rust compiler. 
- **Validation Rule**: The version of the `@claw/cli` NPM package (i.e. `1.2.0`) MUST be intrinsically hard-coded to download the exact matching GitHub Release tag (`v1.2.0`). We cannot allow a floating "latest" binary to be downloaded, as this guarantees a desync between the TypeScript Runtime Engine and the Rust AST structural hashes.

### 4.3 CI/CD & End-to-End Testing (`specs/17-Phase6...`)
- **Impact**: The time required to execute `claw test` in remote environments (like Vercel, AWS CodeBuild, or GitHub Actions) drops dramatically from **~4 minutes** (compiling standard Rust deps) down to **~3 seconds** (downloading a 10MB pre-compiled binary).
- We must update `PRODUCTION.md` and test-runner documentation to use the `curl`/`npm` installation paths rather than `cargo install`.

### 4.4 Shell Installers (Optional Downstream)
- We will eventually host an installation endpoint at `https://claw.sh/install.sh` for users explicitly bypassing NPM (e.g. strict Python environments). This shell script replicates the exact same logic as the NPM `postinstall.js` script utilizing `curl` and `grep / uname`.
