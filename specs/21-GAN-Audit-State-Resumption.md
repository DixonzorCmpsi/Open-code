# Phase 8 GAN Audit: Checkpointing, Resumption, and State Mutations

In this GAN (Generative Adversarial Network) Audit, **The Breaker** (Systems/Security Architect) challenges **The Maker** (Lead Execution Engineer) regarding the structural guarantees of the Claw Gateway's Event Sourcing and Checkpoint OS (`specs/07-OpenClaw-OS.md`).

---

## 1. The Non-Deterministic Loop Attack

**Breaker (The Attacker):**
> "Your Gateway claims to be a deterministic event-sourcing engine. Let's say my `.claw` script has a `for` loop that iterates over an array of 50 URLs generated by an LLM `SearchAgent`.
> The OS crashes on URL #30. When the gateway reboots, it attempts to resume the `session_id`. However, LLMs are non-deterministic! If it re-evaluates the `SearchAgent` node to get the array, it might return 40 totally different URLs. The execution graph is now hopelessly corrupted because loop index 30 no longer points to the original data."

**Maker (The Defender):**
> "You fundamentally misunderstand Event Sourcing. The Gateway doesn't re-evaluate completed nodes."

**Resolution (BREAKER YIELDS - SPEC CLARIFICATION):**
*Implementation Update:* Every AST expression node (e.g., `Node-4-Expr-2`) is assigned a monotonically increasing execution ID during traversal. When the original `SearchAgent` completes, its exact JSON output is committed to SQLite/Redis. Upon resumption, the Gateway replays the Execution Graph. When it hits the `SearchAgent` node, it reads the *cached result* from the checkpoint database instead of calling the LLM. Non-determinism is entirely neutralized because historical states are immutable.

---

## 2. The Schema Evolution Mismatch (AST Drift)

**Breaker (The Attacker):**
> "Fine, the graph replays exactly. But what happens across deployments?
> Developer deploys `v1.0` of `pipeline.claw`. A massive workflow hits a `pause_for_human` break, suspending its state in Redis.
> The Developer then modifies the `CompetitorData` type in `.claw`, adds a `revenue: int` required field, recompiles (`v1.1`), and deploys the new SDK/AST to production. 
> The suspended `v1.0` session wakes up and tries to resume. But the new AST requires `revenue: int`, which doesn't exist in the old Redis checkpoint! The TypeBox Bouncer violently crashes the workflow mid-flight in production."

**Maker (The Defender):**
> "This is a classic distributed systems problem. Data at rest vs. Schema at runtime. We cannot magically infer missing fields."

**Resolution (MAKER YIELDS - GATEWAY UPGRADE):**
*Implementation Fix:* We strictly enforce **AST Hashing Checkpointing**. 
1. `session_id` records in the database MUST include the `OPENCLAW_AST_HASH` they were created under.
2. The Gateway can only resume a checkpoint if the runtime `AST_HASH` perfectly matches the checkpoint's hash. 
3. *Downstream Impact:* To prevent blocking, `claw-gateway` must support "Drain Mode"—running both the old AST and the new AST side-by-side in memory, routing resumed sessions to their original AST definitions until they naturally complete, while new sessions hit the latest deployment.

---

## 3. The Poisoned Checkpoint Injection

**Breaker (The Attacker):**
> "Your TypeBox 'Bouncer' rigorously guards the entry points where the LLM talks to the system. But what about the Database? 
> You said you use Redis/SQLite for the Checkpoint store. If an attacker gains internal read/write access to the Redis cache, they can manually mutate the serialized JSON state of a suspended workflow. When the Gateway resumes the workflow, it loads the poisoned state directly into memory, entirely bypassing your TypeBox Bouncer because 'it already passed validation before it was saved.' You just allowed remote code execution or data corruption."

**Maker (The Defender):**
> "Zero-trust environments require zero-trust databases. We shouldn't implicitly trust the JSON restored from our own checkpoints."

**Resolution (MAKER YIELDS - OS HARDENING):**
*Implementation Fix:* The TypeBox Bouncer validation must be decoupled from the LLM network layer. 
1. During `Resume` execution, as the AST replays historical node outputs from the Checkpoint Database, **EVERY** restored payload must pass through the TypeBox schema validator *again* before entering the interpreter's memory stack.
2. If restored state fails validation (due to tampering, Redis corruption, or manual database edits), the OS instantly halts the resume operation with a `CorruptedCheckpointError` and throws the workflow into a `FAULTED` dead-letter queue.

---

## 4. The Infinite Retry Black Hole

**Breaker (The Attacker):**
> "Your pipeline says an LLM error or schema mismatch triggers an automatic retry. 
> If `Model X` is fundamentally incapable of returning the complex nested JSON your TypeBox demands, it will fail. Your Gateway throws an error, the catch block catches it, or the OS just infinitely retries the node. You're going to burn through a user's $1,000 monthly OpenAI limit overnight because a stuck loop keeps slamming the API every 500ms hoping for a better formatted response."

**Maker (The Defender):**
> "The compiler specs mention retry limits, but the Gateway spec currently lacks a circuit breaker definition for hard schema degradations."

**Resolution (MAKER YIELDS - FINANCIAL SECURITY GUARDRAIL):**
*Implementation Fix:* Update `specs/07-OpenClaw-OS.md` to define a hard-coded, non-configurable **Circuit Breaker**:
- Max continuous LLM schema degradation retries = 3. 
- After 3 consecutive TypeBox unparseable failures on a single node execution attempt, the node fatally faults the workflow. The Gateway will not allow infinite LLM spinning under any circumstance.
- A `claw.json` property `max_cost_per_session` will be tracked by standardizing token estimation before firing the requests.

---

### Audit Conclusion
The Event Sourcing architecture is powerful but dangerously rigid. The ability to pause and resume agent workflows is the "killer feature" of Claw, but it introduces massive surface areas for AST Version Drift, Data Corruption, and Non-Deterministic re-evaluations. 

By enforcing **Immutable Cache Replays**, **AST Hash Bindings**, **Zero-Trust Restorations**, and **Financial Circuit Breakers**, the Claw Runtime OS becomes legitimately enterprise-ready and secure against both distributed deployment chaos and malicious internal attacks.
