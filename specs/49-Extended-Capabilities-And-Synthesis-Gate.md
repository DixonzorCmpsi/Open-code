# Spec 49: Extended Capabilities and Synthesis Sandbox Gate

**Status:** ACTIVE — filed 2026-03-21.
**Depends on:** Spec 03 (Grammar), Spec 04 (AST), Spec 32 (Synthesis Pipeline), Spec 34 (Advanced Tool Patterns), Spec 36 (Synthesis Repair Loop), Spec 48 (Canonical Architecture).
**Ripple effects:** Spec 01 §3, Spec 03 §1 (using_expr), Spec 04 (UsingExpr), Spec 33 §2.1 (capability field), Spec 36 §3 (capability hint), Spec 46 §4 (capability research), Spec 48 §6.3 (gap matrix — closes two entries).

---

## 0. Motivation

Two capability gaps and one security gap exist in the current synthesis pipeline:

1. **No macOS GUI automation** — `using: bash` can launch apps but cannot control them. Controlling CapCut, Xcode, Finder, or any macOS app requires AppleScript (`osascript`). No DSL path exists today.

2. **No visual computer interaction** — For apps with no scripting API, the only path is vision + input simulation (Claude computer-use). No DSL path exists today.

3. **No isolation on synthesized code before execution** — After `claw synthesize` writes `generated/tools/<Name>.ts`, it is immediately executed by the workflow. Synthesized code runs with full ambient permissions: filesystem, network, environment. A buggy or adversarial synthesis can cause side effects before any test gate catches it. The Synthesis Sandbox Gate closes this by running synthesized code in an isolated subprocess with the contract test inputs, verifying the output shape before the tool is considered live.

## 1. `using: applescript` — macOS GUI Automation

### 1.1 Concept

`using: applescript` synthesizes TypeScript that drives macOS applications via `osascript`. This covers:
- Launching and focusing applications
- Menu navigation, button clicks, text input
- Reading application state (window titles, field values)
- Inter-app automation (Finder → CapCut → export pipeline)

AppleScript executes as the user's own process with the user's permissions. It does not require elevated privileges for most GUI automation.

### 1.2 DSL Grammar

The `using:` rule in `tool_decl` (Spec 03 §1) gains two new terminals:

```
// Old:
using_expr = { "fetch" | "playwright" | "bash" | ("mcp" ~ "(" ~ string_lit ~ ")") | ("baml" ~ "(" ~ string_lit ~ ")") }

// New (additive — no existing rules removed):
using_expr = {
    "fetch"
  | "playwright"
  | "bash"
  | "applescript"
  | "computer"
  | ("mcp"  ~ "(" ~ string_lit ~ ")")
  | ("baml" ~ "(" ~ string_lit ~ ")")
}
```

Example tool declaration:

```claw
tool FocusCapCut() -> AppState {
    using: applescript
}

tool ExportCapCutVideo(project_name: string) -> ExportResult {
    using:       applescript
    description: "Focuses CapCut, opens the named project, and triggers export."
    test {
        input:  { project_name: "demo_reel" }
        expect: { success: true, export_path: !empty }
    }
}
```

### 1.3 AST Change (Spec 04)

`UsingExpr` in `src/ast.rs` gains one new variant:

```rust
pub enum UsingExpr {
    Fetch,
    Playwright,
    Bash,
    AppleScript,   // NEW
    Computer,      // NEW — see §2
    Mcp(String),
    Baml(String),
}
```

No other AST changes. `ToolDecl.using: Option<UsingExpr>` is unchanged.

### 1.5 Capability Research (injected into skill spec by `skill_spec.rs`)

```
## Capability: `applescript` — Implementation Patterns

Use `osascript` (macOS built-in) via `execFile`. Never use `exec` with shell interpolation.

**Pattern A — inline script:**
```typescript
const script = `
  tell application "CapCut"
    activate
  end tell
`;
const { stdout, stderr } = await _execFile('osascript', ['-e', script], { timeout: 15_000 });
return { ... } satisfies AppState;
```

**Pattern B — multi-line with data extraction:**
```typescript
const script = [
  '-e', 'tell application "Finder"',
  '-e', `set f to POSIX file "${safePath}"`,
  '-e', 'open f',
  '-e', 'end tell',
];
const { stdout } = await _execFile('osascript', script, { timeout: 10_000 });
```

**Security rules:**
- NEVER interpolate raw user input into a script string — use Pattern B (separate `-e` flags)
  so each component is a separate argument, not shell-expanded.
- Validate any path argument: strip shell metacharacters before using in a script.
- Set explicit timeouts (default 15s for GUI operations).

**Platform check:**
- `osascript` only exists on macOS. If the tool must run cross-platform, check at runtime:
  `if (process.platform !== 'darwin') throw new Error('applescript tools require macOS')`.

**Common targets:** Finder, Safari, Chrome, CapCut, Final Cut Pro, Xcode, Numbers, Pages,
Mail, Calendar, Reminders. Use `System Events` for apps without AppleScript dictionaries.
```

### 1.6 Semantic Validation Rules

- `using: applescript` on a tool compiles normally on all platforms.
- The compiler emits `W-AS01: AppleScriptNonMacOS` when `cfg!(not(target_os = "macos"))` OR when `CLAW_PLATFORM` env var is set to a non-macOS value at build time. This is a warning, not an error — cross-platform `.claw` files are valid.
- The generated TypeScript includes a runtime platform check (see §1.5) that throws clearly on non-macOS.
- `using: applescript` tools may declare `secrets {}` — injected as env vars, same as `using: bash`.

## 2. `using: computer` — Visual Computer Interaction

### 2.1 Concept

`using: computer` synthesizes TypeScript that uses a vision model to interact with the screen. The synthesized code:
1. Captures a screenshot (via `screencapture` on macOS or `import` on Linux)
2. Sends the screenshot + goal to a computer-use capable model (Anthropic `claude-opus-4-6` by default, configurable)
3. Executes the model's action instructions (click, type, scroll, keypress) via `osascript` or `xdotool`
4. Loops until the model signals completion or a step limit is reached
5. Captures final state and returns typed output

**Critical distinction:** Unlike all other `using:` types, `using: computer` **calls an LLM at workflow execution time** — it is not purely deterministic. The type system must mark this as `LLM_AT_RUNTIME`. See §2.5.

### 2.2 DSL Grammar

Same grammar extension as §1.2. The `computer` terminal is a keyword with no payload:

```claw
tool ClickExportButton(window_title: string) -> ExportResult {
    using:       computer
    description: "Locates the Export button in the named window and clicks it."
    test {
        input:  { window_title: "My Project" }
        expect: { success: true }
    }
}
```

To override the vision model (default: synthesizer's client, or `claude-opus-4-6`):

```claw
tool ReadScreenText(region: ScreenRegion) -> TextResult {
    using:   computer
    client:  VisionModel    // optional — references a declared client block
}
```

This requires a `client:` property on tool declarations — see §2.3.

### 2.3 New Tool Property: `client:` (computer-use model override)

`using: computer` tools may declare an optional `client:` property to override the default vision model:

```
tool_prop = {
    ("using"       ~ ":" ~ using_expr)    |
    ("invoke"      ~ ":" ~ invoke_expr)   |
    ("description" ~ ":" ~ string_lit)    |
    ("client"      ~ ":" ~ identifier)    |   // NEW — only valid on using:computer tools
    ("secrets"     ~ "{" ~ identifier+ ~ "}") |
    test_block                            |
    examples_block
}
```

**Semantic rule:** `client:` on a tool declaration is only valid when `using: computer` is also declared. Any other `using:` type with a `client:` property → `E-CU02: ClientOnNonComputerTool` compile error.

The referenced `client` must be declared in the same document → existing `UndefinedClient` error applies.

**AST change:** `ToolDecl` gains an optional `computer_client: Option<String>` field (populated only for `using: computer` tools):

```rust
pub struct ToolDecl {
    pub name:            String,
    pub arguments:       Vec<TypeField>,
    pub return_type:     Option<DataType>,
    pub invoke_path:     Option<String>,
    pub using:           Option<UsingExpr>,
    pub synthesizer:     Option<String>,
    pub test_block:      Option<TestBlock>,
    pub secrets:         Vec<String>,
    pub computer_client: Option<String>,   // NEW — only used when using = Computer
    pub span:            Span,
}
```

### 2.5 LLM-at-Runtime Marker

`using: computer` tools are the only synthesis-path tools where an LLM is called at execution time. The compiler marks these with a new semantic annotation:

- `ToolDecl` semantic analysis (Pass 3): if `using = Computer`, set `tool_is_llm_at_runtime = true`.
- Warning `W-CU01: ComputerToolIsLLMAtRuntime` — always emitted (informational, not suppressable):
  ```
  warning: tool `ClickExportButton` uses computer-use and calls a vision model at runtime.
           This tool is non-deterministic. Execution cost and latency are unbounded.
           Ensure ANTHROPIC_API_KEY (or configured client key) is set before `claw run`.
  ```
- The generated skill spec includes a **## Runtime Cost** section with token estimate guidance.

### 2.6 Capability Research (injected into skill spec by `skill_spec.rs`)

```
## Capability: `computer` — Implementation Patterns

The computer-use loop runs a vision model at execution time. It is non-deterministic
and incurs LLM API cost on every invocation.

**Full pattern:**
```typescript
import { execFile } from 'node:child_process';
import { promisify } from 'node:util';
const _execFile = promisify(execFile);

const MODEL = 'claude-opus-4-6';
const MAX_STEPS = 10;

async function takeScreenshot(): Promise<string> {
  const path = `/tmp/claw-screenshot-${Date.now()}.png`;
  await _execFile('screencapture', ['-x', path]);
  const data = await fs.promises.readFile(path);
  return data.toString('base64');
}

async function executeAction(action: { type: string; [k: string]: unknown }): Promise<void> {
  // click: use osascript System Events
  // type: use osascript keystroke
  // scroll: use osascript scroll wheel
  // key: use osascript key code
}
```

**Security rules:**
- ALWAYS cap iterations with `MAX_STEPS` (default 10) — unbounded loops are a cost DoS risk.
- NEVER pass raw user-controlled strings as the vision goal — sanitize before use.
- The ANTHROPIC_API_KEY must be set in env — never hardcode.
- On non-macOS: use `import` (ImageMagick) for screenshots, `xdotool` for input simulation.

**Step budget:** Each vision call costs ~500–2000 tokens depending on screenshot complexity.
At 10 steps with claude-opus-4-6, budget for ~5K–20K tokens per invocation.
```

### 2.7 Semantic Validation Rules

- `using: computer` is valid on any platform at compile time.
- Runtime check: `screencapture` (macOS) or `import` (Linux) must be available.
- `W-CU01` is always emitted — see §2.5.
- If no `client:` override and no `synthesizer {}` declared → default to `claude-opus-4-6` with `ANTHROPIC_API_KEY`.
- If `client:` override references a `provider = "local"` client → `E-CU01: ComputerToolRequiresVisionModel` compile error (local models do not support computer-use actions).
- `using: computer` tools may declare `secrets {}` for additional env vars.
- `using: computer` + `using: bash` or `invoke:` on the same tool → existing mutual exclusion error applies.

## 3. Synthesis Sandbox Gate

### 3.1 Concept

The current synthesis pipeline writes synthesized TypeScript and immediately runs `vitest` in the host process. This has two problems:

1. **No isolation** — synthesized code runs with full host permissions. A bug or adversarial synthesis can write files, make network requests, or corrupt state before any test catches it.
2. **No smoke test before vitest** — `vitest` runs the full test suite. If the synthesized code throws on the first line, the diagnostic is buried in vitest output. A subprocess smoke test gives a clean signal.

The **Synthesis Sandbox Gate** is a new Stage 2.5 inserted between synthesis and vitest:

```
Stage 2: synthesize (OpenCode writes generated/tools/<Name>.ts)
    │
    ▼ Stage 2.5: sandbox gate  ← NEW
        tsc --noEmit (type-check)
        spawn isolated subprocess with test{} inputs
        verify output shape matches declared return type
        on fail → Spec 36 repair loop
    │
    ▼ Stage 3: vitest contract tests
    │
    ▼ Stage 4: bundle
```

### 3.2 DSL Configuration

Optional `sandbox_gate {}` block on `synthesizer {}`:

```claw
synthesizer DefaultSynth {
    client      = LocalQwen
    temperature = 0.1
    retry {
        max_attempts: 3
        strategy:     repair
    }
    sandbox_gate {
        enabled:    true      // default: true for bash/applescript/computer; false for fetch/playwright/mcp/baml
        timeout_ms: 8000      // default: 8000ms
        network:    none      // none | host — default: none for bash/applescript/computer; host for fetch/playwright
    }
}
```

**Defaults when `sandbox_gate {}` is absent** (applied per tool capability):

| Capability | `enabled` default | `network` default | Rationale |
|---|---|---|---|
| `bash` | `true` | `none` | Bash tools should not need network; isolation prevents fs writes |
| `applescript` | `true` | `none` | GUI tools should not make network calls |
| `computer` | `true` | `host` | Vision model call requires outbound HTTPS |
| `fetch` | `false` | `host` | Fetch tools require network; gate would require live endpoints |
| `playwright` | `false` | `host` | Browser automation requires network + display |
| `mcp` | `false` | `host` | MCP depends on live server |
| `baml` | `false` | `host` | BAML calls live LLM |

### 3.3 Generated Gate Runner

For each tool with a `test {}` block and an enabled gate, `claw build` generates:

```typescript
// generated/__tests__/gates/<ToolName>.gate.ts
// AUTO-GENERATED — do not edit. Runs in isolated subprocess during claw synthesize.
// Exit 0 = gate passed. Exit 1 = gate failed.

import { <ToolName> } from '../../tools/<ToolName>.js';
import type { <ReturnType> } from '../../types.js';

const TIMEOUT_MS = 8000;

const timer = setTimeout(() => {
  process.stderr.write('gate timeout\n');
  process.exit(2);
}, TIMEOUT_MS);

async function run(): Promise<void> {
  // Test inputs from test{} block
  const input = { <test_inputs> };

  let result: <ReturnType>;
  try {
    result = await <ToolName>(input);
  } catch (err: unknown) {
    const msg = err instanceof Error ? err.message : String(err);
    process.stderr.write(`gate error: ${msg}\n`);
    process.exit(1);
  }

  // Shape check: output must be a non-null object
  if (typeof result !== 'object' || result === null) {
    process.stderr.write(`gate shape: expected object, got ${typeof result}\n`);
    process.exit(1);
  }

  // Field presence check from declared return type
  const missing = [<required_fields>].filter(k => !(k in result));
  if (missing.length > 0) {
    process.stderr.write(`gate fields: missing fields: ${missing.join(', ')}\n`);
    process.exit(1);
  }

  // Quantitative checks from expect: block
  <expect_assertions>

  clearTimeout(timer);
  process.stdout.write(JSON.stringify(result) + '\n');
  process.exit(0);
}

run();
```

The `<expect_assertions>` block mirrors the tool's `expect:` entries:
```typescript
// For each expect: { field: >= value }
if (!(result.<field> >= <value>)) {
  process.stderr.write(`gate expect: result.<field> >= <value> failed (got ${result.<field>})\n`);
  process.exit(1);
}
```

### 3.4 Gate Execution in `run_synthesize()`

In `src/bin/claw.rs`, `synthesize_tool()` gains a gate step after synthesis and before marking success:

```rust
// After OpenCode writes the tool file and SYNTHESIS_COMPLETE is detected:

// Step 1: tsc --noEmit (fast type check)
let tsc_result = run_tsc_check(tool, project_root)?;
if !tsc_result.success {
    last_error = Some(format!("TypeScript compile error:\n{}", tsc_result.stderr));
    continue; // retry
}

// Step 2: Sandbox gate (if enabled for this capability)
if should_run_gate(tool, &document) {
    let gate_result = run_sandbox_gate(tool, project_root, gate_config)?;
    if !gate_result.success {
        last_error = Some(format!("Sandbox gate failed:\n{}", gate_result.stderr));
        continue; // retry → Spec 36 repair loop
    }
}
// Both passed → synthesis complete
return Ok(());
```

### 3.5 Gate Subprocess Isolation

The gate subprocess is spawned with:

```rust
Command::new("node")
    .arg("--input-type=module")
    .arg(gate_script_path)
    .env_clear()
    .env("PATH", restricted_path)      // only /usr/bin:/usr/local/bin, no user PATH
    .env("CLAW_SANDBOX", "1")
    .env("NODE_PATH", node_modules_path)
    .envs(tool_secrets_from_host_env)  // only the declared secrets, nothing else
    // network: if network=none, spawn inside a network namespace (macOS: use sandbox-exec)
    .timeout(Duration::from_millis(gate_config.timeout_ms))
    .output()?
```

**macOS network isolation for `network: none`:**
On macOS, gVisor is not available. Network isolation uses `sandbox-exec` with a Seatbelt profile that denies all outbound connections:

```
(version 1)
(deny default)
(allow file-read*)
(allow process-exec)
(deny network*)
```

Spawned as: `sandbox-exec -p '(version 1)(deny default)(allow file-read*)(allow process-exec)(deny network*)' node gate.js`

**Security model for the gate:**
- The gate is **defence-in-depth for accidental bugs**, not a hard security boundary against adversarial synthesis.
- An adversarial LLM could read files — the PATH restriction and env_clear limit the blast radius.
- The gate is spawned by the user who owns the project — it has the user's filesystem read permissions.
- For production environments where synthesis happens in CI, use `runtime = docker` in a `sandbox {}` block (Spec 34) for the actual workflow execution — the gate is a dev-time check, not a runtime security primitive.

### 3.6 Gate Skip Conditions

The gate is **skipped** (only tsc type-check runs) when:
- The tool has no `test {}` block (no inputs to drive the gate)
- The capability default is `enabled: false` (see §3.2 table) and `sandbox_gate {}` does not override
- `CLAW_SKIP_GATE=1` env var is set (emergency escape hatch for CI debugging)
- The gate runner file does not exist (e.g., tool has only `expect: { field: !empty }` — shape check only runs)

When skipped, `claw synthesize` logs:
```
  ⚠ gate skipped for <ToolName> (no test{} block) — shape check only
```

### 3.7 Gate Failure → Spec 36 Repair Loop

Gate failure is treated identically to a vitest failure in the Spec 36 repair loop. The error context passed to the repair prompt includes:

```
## Gate Failure (Stage 2.5)
The synthesized code passed TypeScript compilation but failed the sandbox gate.

Gate error:
<stderr from the gate subprocess>

Gate inputs (from test{} block):
<test inputs as JSON>

Fix the implementation. The gate runner checks the following:
<list of field checks and expect assertions>
```

This gives the repair LLM full context: the code, the inputs, the output shape expected, and exactly what assertion failed.

### Layer 4 — Coherence Check

- `using: applescript` is purely additive — the existing `using: bash` rules are not changed.
- `using: computer` is the only `using:` type that calls an LLM at runtime. This is explicitly marked with `W-CU01` and documented in the generated TypeScript scaffold. It does not break the Spec 32 determinism guarantee for other tool types.
- The Synthesis Sandbox Gate is additive to the existing pipeline — it is a Stage 2.5 insert. Spec 32 Stage 3 (vitest) is unchanged; it runs after the gate. Spec 36 (repair loop) triggers on gate failure using the same repair mechanism it already uses for vitest failures.
- `client:` on tool declarations is scoped to `using: computer` only. It does not affect agent `client` declarations or synthesizer `client` declarations. The semantic rule `E-CU02` prevents misuse.

## 6. Non-Goals

- **Windows GUI automation** (`using: winscript` or `using: powershell`) — out of scope. Add as a future capability.
- **Linux GUI automation** (`using: xdotool`) — out of scope. Add as a future capability. The `using: computer` path (visual) works on Linux.
- **Container-level gate isolation** (gVisor/Docker for the synthesis gate) — out of scope. The gate uses subprocess + macOS Seatbelt for dev-time checks. Spec 34's `sandbox {}` block is the production isolation primitive.
- **Gate network isolation on Linux** — `sandbox-exec` (Seatbelt) is macOS-only. Linux gate runs without network isolation. Add `unshare --net` support in a future spec.
- **`using: computer` on non-Anthropic providers** — the computer-use action format (`click`, `type`, `scroll`) is Anthropic-specific. Cross-provider support is a future spec.
- **Cost tracking for computer tools** — token count and USD cost per run is not tracked in the artifact. Future: `claw run --cost-report`.
